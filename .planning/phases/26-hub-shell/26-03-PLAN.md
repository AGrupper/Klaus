<objective>
Build the hub authentication boundary: Google Sign-In allowlisted to Amit's account only → an itsdangerous-signed httpOnly session cookie → a `require_hub_session` FastAPI dependency that every `/api/*` route depends on. This is the single control boundary protecting a live, write-capable command channel into Klaus.

Purpose: HUB-01 (full) — the hub is on the public internet and the chat can drive Gmail/calendar writes and LLM spend, so auth is non-optional. Per D-01 the session is effectively permanent (365-day cookie, silently refreshed on each visit); per D-02 a bumpable `session_version` counter (added to `UserProfileStore` in 26-02) gives sign-out-everywhere without a Firestore-backed session store. The auth path is brand new and MUST NOT weaken the existing OIDC on `/cron/*` and `/internal/*` (HUB-04).

Output: `interfaces/hub_auth.py` (verify + sign + dependency), three `/api/auth/*` routes in `web_server.py`, the `SignInPage.tsx` + auth store/api on the frontend, and `tests/test_hub_auth.py` flipped from Wave 0 skips to real assertions.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| internet → `/api/auth/google` | Untrusted GIS ID token crosses here; verified server-side via `verify_oauth2_token` before any cookie issues |
| browser cookie → `/api/*` | The signed session cookie is the sole credential for every authed hub route |
| `/api/auth/revoke-all` → UserProfileStore.session_version | Server-side control surface for sign-out-everywhere (D-02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26-03-01 | Spoofing | Forged/replayed GIS ID token at `/api/auth/google` | mitigate | `id_token.verify_oauth2_token` validates signature + `exp` + audience (GOOGLE_OAUTH_CLIENT_ID); reject `email_verified` false; GIS tokens expire in 1h |
| T-26-03-02 | Elevation of Privilege | Any non-Amit verified Google account signing in | mitigate | Allowlist check `email == HUB_ALLOWED_EMAIL` in BOTH `verify_google_id_token` (sign-in) and `verify_session_cookie` (every request) → 403 |
| T-26-03-03 | Information Disclosure | XSS in SPA reading the session cookie | mitigate | `httponly=True` (JS cannot read it); cookie value never logged (redacted prefix only) |
| T-26-03-04 | Tampering | CSRF — cross-site POST to a cookie-authed `/api/*` route | mitigate | `samesite="strict"` primary control; same-origin architecture (no CORS) eliminates cross-origin request surface |
| T-26-03-05 | Elevation of Privilege | Stolen cookie on a lost phone (D-02) | mitigate | `session_version` counter in UserProfileStore; `/api/auth/revoke-all` bumps it; every old cookie fails the version check → 401 |
| T-26-03-06 | Tampering | Forged/tampered session cookie payload | mitigate | `itsdangerous.TimestampSigner` HMAC-SHA256 signature; tamper → `BadSignature` → 401 |
| T-26-03-07 | Denial of Service | `HUB_SESSION_SECRET` unset → fail-open signing | mitigate | Refuse-all-on-unset-env (500 "Server misconfigured") — never sign with an empty secret |
| T-26-03-08 | Elevation of Privilege | New auth path weakening existing OIDC on `/cron/*`,`/internal/*` | mitigate | `require_hub_session` is a NEW dependency on `/api/*` only; `_verify_cron_request`/`_verify_healthkit_request` untouched; SPA mount stays last (HUB-04) |
</threat_model>
