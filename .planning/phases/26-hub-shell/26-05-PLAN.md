<objective>
Build the hub chat backend: a dedicated Cloud Tasks full-CPU path so hub messages run the agent turn exactly like Telegram, sharing one continuous Firestore conversation (one Klaus).

Purpose: CHAT-01/02 + the dispatch half of CHAT-03/04. Per D-09 (locked requirement CHAT-02 wins over the design-spec prose) hub messages go through a DEDICATED `/internal/process-hub-message` endpoint on the same full-CPU Cloud Tasks pattern as `/internal/process-update` — NEVER a Starlette BackgroundTask (CLAUDE.md invariant: background tasks get CPU-throttled after the response). The hub keys the shared `FirestoreConversationStore` on `telegram_user_id` (added to `UserProfileStore` in 26-02) so the hub and Telegram are one history (CHAT-01). `POST /api/chat` appends the user message + enqueues; `GET /api/chat/messages` returns the recent window for fast first paint + scroll-up history (D-08, RESEARCH Pattern 7). The frontend optimistic-send + 2.5s polling + unread badge (CHAT-03/04) consume these endpoints in 26-08.

Output: `enqueue_hub_message` in `core/task_dispatch.py`, the three routes in `web_server.py`, and `tests/test_hub_chat.py` + the `enqueue_hub_message` test extension flipped to real assertions.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → `POST /api/chat` | A live, write-capable command channel into Klaus (Gmail/calendar writes, LLM spend); gated by `require_hub_session` (the allowlist+session is THE control boundary) |
| Cloud Tasks → `/internal/process-hub-message` | OIDC-gated internal endpoint; only the Cloud Tasks SA may invoke it |
| hub user message → shared FirestoreConversationStore | Same history as Telegram; one Klaus — no fork |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26-05-01 | Elevation of Privilege | Unauthenticated internet user driving Klaus's tools (calendar/Gmail writes, LLM spend) via `/api/chat` | mitigate | `Depends(require_hub_session)` (allowlist + signed cookie from 26-03) is the control boundary; `/api/chat` 401s without it |
| T-26-05-02 | Elevation of Privilege | Open `/internal/process-hub-message` accepting arbitrary turns from the internet | mitigate | `_verify_cron_request` OIDC gate (same SA + audience as /internal/process-update); `test_internal_process_hub_message_oidc_gated` |
| T-26-05-03 | Denial of Service | Agent turn run as a Starlette BackgroundTask → CPU-throttled after response → 18-min replies | mitigate | Turn runs INSIDE the tracked Cloud Tasks request via `asyncio.to_thread(handle_message)`; no `background_tasks.add_task` on the hub path (D-09 / CLAUDE.md invariant) |
| T-26-05-04 | Tampering | Malformed/oversized chat body injecting into the agent loop | mitigate | Pydantic model on `/api/chat` (non-empty, max length — ASVS V5); `{content, user_id}` JSON only on the internal endpoint |
| T-26-05-05 | Tampering | Cloud Tasks retry on a stalled LLM call → duplicate Klaus replies | mitigate | Same `_DISPATCH_DEADLINE_SECONDS=540` + `LLM_TIMEOUT_SECONDS=120` as the Telegram path (Pitfall 7) — no new exposure |
| T-26-05-06 | Information Disclosure | `DatetimeWithNanoseconds` 500 leaking a trace on `/api/chat/messages` | mitigate | `_jsonsafe_doc` on the messages response (Pitfall 4) |
