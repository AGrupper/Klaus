<closed_sets>
benchmark_due eligibility (D-02): set True only when the active block (date-range resolved) is one of Blocks 1-3, end_date - today within 3 days, AND NOT the Block-4 race (detect by label "Race Specificity → Taper → Race Week" OR by end_date 2026-10-10). Block 4 NEVER gets benchmark_due.
Gate thresholds (D-07): defer if (hrv_overnight/hrv_baseline) < 0.70 OR acwr_ratio > 1.2.
Three benchmark states (exactly one when benchmark logic fires):
  "benchmark_window_open": benchmark_due AND gate passes → prompt all 5 facets
  "benchmark_deferred": benchmark_due AND gate fails AND today <= end_date → defer + numeric reason (D-08)
  "benchmark_stale": benchmark_due AND today > end_date (window closed still red) → one prompt w/ tested-under-fatigue caveat (D-09)
Gate-unknown fallback: if Garmin/ACWR data missing → treat as PASS (err toward prompting, per RESEARCH Environment Availability).
Send-path requirement (CORRECTNESS — covers BLOCK-02 SC "benchmark fires within 3 days"): the 21:30 cron currently early-returns (lines 136-139) when there is no weather/overload/travel alert, skipping compose+send. A pending benchmark state is most often the ONLY thing to say on a deload night. The benchmark evaluation must therefore happen BEFORE that early-return decision (so a benchmark state can keep the cron in the send path), OR the early-return condition must be widened to "no alerts AND no benchmark state". Either way: a benchmark_window_open/deferred/stale state on a night with zero other alerts MUST still produce a sent message.
