# Phase 19.1: HealthKit Nutrition Bridge — Context

**Gathered:** 2026-05-28
**Status:** Ready for planning
**Parent:** Phase 19 (Training Awareness & Nutrition Coaching) — held open pending this gap-closure phase

<domain>
## Phase Boundary

Bridge Apple HealthKit nutrition data (where Lifesum writes on iOS) into Klaus's existing `MealStore`, so the autonomous tick can act on real meal data and Phase 19 SC #2 closes.

**In scope:**
- iOS Shortcuts that read HealthKit nutrition samples and POST to a new Klaus webhook
- New `/cron/healthkit-sync` FastAPI route with shared-secret bearer auth
- Normalization layer that maps HealthKit's `HKQuantityType` nutrition fields + `HKMetadataKeyMealTime` into the existing `MealStore` document shape
- Reuse `MealStore` as a multi-source store (gain `source="healthkit"` writer alongside the existing `source="google_fit"` Android path)
- Operator validation tool + Pydantic schema lock + heartbeat staleness monitoring + operator runbook
- Documentation updates: `docs/SELF.md`, `docs/DEPLOYMENT.md`, new `docs/healthkit_shortcut.md`

**Out of scope (deferred to other phases or stays as-is):**
- Apple Watch workout sync (workouts stay on Garmin per Phase 19 contract)
- HRV / sleep / body-battery from HealthKit (Garmin already covers these; iOS HealthKit not a second source)
- Removal of the Google Fit nutrition path (kept, marked legacy)
- Personalized meal-critique thresholds (v3.0 milestone discipline: plumbing only)
- New tick-brain triggers — existing `_is_empty_signals` already treats non-empty `meals_since_last_tick` as a triggering signal (NUTR-04, Phase 19-04)
- Backfill of historical HealthKit nutrition data

This phase is plumbing-only, gap-closure for Phase 19's SC #2. It does NOT add new judgment behaviour to Klaus — it just feeds real data into a pipeline that's already wired and eval-tested.

</domain>

<decisions>
## Implementation Decisions

### Transport (iPhone → Klaus)

- **D-01:** **Transport mechanism = iOS Shortcuts** (free, native). Health Auto Export and other paid bridges declined — long-term-efficiency preference.
- **D-02:** **Primary trigger = iOS Personal Automation "When Lifesum is Closed"**. Action chain: `Find Health Samples Where (Date is in the last 2 hours, type ∈ Dietary Energy / Protein / Carbs / Fat / Fiber / Sugar)` → assemble JSON → `Get Contents of URL` (POST to Klaus webhook). Toggle **Run Without Asking** so the automation fires silently (default since iOS 15.4 for app triggers).
- **D-03:** **Sample window per fire = last 2 hours**. Trade-off acknowledged: if the close-trigger misses (phone locked at the moment Lifesum backgrounds), a 2h window may not catch the meal on the next fire. The daily catch-up (D-04) rescues those cases within 24h.
- **D-04:** **Safety net = second iOS Personal Automation at 23:55 daily with 24h lookback window**. Idempotent on the Klaus side because `MealStore.upsert` dedupes by `source_id = "healthkit:{HKObjectUUID}"` — re-sending the same samples 100 times still produces one Firestore doc per HealthKit object.
- **D-05:** **iPhone failure handling = notify-on-fail**. The Shortcut adds a "Show Notification" action in the HTTP error branch — Amit sees a lock-screen notification "Klaus meal push failed" if the webhook returns 5xx or times out. **No automatic per-fire retry**; the 23:55 catch-up handles eventual consistency.

### Webhook auth & secrets

- **D-06:** **Auth model = static shared-secret bearer token**. Stored in Secret Manager as `klaus-healthkit-webhook-token`. New helper `_verify_healthkit_request(request)` in `interfaces/web_server.py` checks `Authorization: Bearer <secret>`. Mirrors the Phase-18 `_verify_cron_request` shape but accepts the static token instead of an OIDC JWT (the iPhone has no Google service account).
- **D-07:** **Rotation flow = standard secrets pattern**. `gcloud secrets versions add klaus-healthkit-webhook-token --data-file=-` → Cloud Run redeploy (or `gcloud run services update --update-secrets`) → `gcloud secrets versions disable` the old version → paste the new token into the iOS Shortcut's `Authorization` header.
- **D-08:** **Kill-switch = secret-version disable**. To emergency-disable the bridge in production without redeploying: `gcloud secrets versions disable klaus-healthkit-webhook-token --version=<n>`. All inbound pushes fail auth immediately. Re-enable by adding a new version. No env-flag plumbing.

### Endpoint behaviour & data flow

- **D-09:** **Endpoint = `POST /cron/healthkit-sync`** in `interfaces/web_server.py`. Path uses `/cron/` prefix for consistency even though the driver is not Cloud Scheduler (the prefix is now the convention for "verified webhook from a trusted external client").
- **D-10:** **Pipeline = upsert-only, defer judgment to next autonomous tick**. Handler does: verify auth → parse Pydantic model → for each sample, build `source_id = "healthkit:{HKObjectUUID}"` → call `MealStore.upsert(source_id, normalized_meal)` → return `200 OK` with `{"upserted": N}`. **No inline tick-brain call.** The next `*/20 7-21` autonomous tick picks the meal up via the existing `meals_since_last_tick` trigger (Phase 19-04), runs the eval-tested tick-brain triage, and decides whether to nudge. Worst-case nudge latency: ~20 min after Lifesum close. Preserves all 5 eval fixtures' coverage; zero new judgment paths.
- **D-11:** **Backfill scope = forward-only from deploy day**. No one-time historical pull. The morning recap will be empty on the first morning post-deploy, then populated from day 2 onwards.

### Data shape & normalization

- **D-12:** **`source_id` namespace = `"healthkit:{HKObjectUUID}"`** for HealthKit-sourced entries, mirroring the existing `"{dataStreamId}:{startTimeNanos}"` Google Fit pattern. HealthKit's `HKObject.UUID` is a stable per-sample identifier that Lifesum (or any HealthKit writer) emits — perfect idempotency anchor. No namespace collision with the Google Fit path.
- **D-13:** **`meal_type` inference = HKMetadataKeyMealTime if Lifesum wrote it, else timestamp-bucket heuristic**. The iOS HealthKit constant `HKMetadataKeyMealTime` carries an integer enum: `1 = breakfast`, `2 = lunch`, `3 = dinner`, `4 = snack`. Lifesum often (not always) populates this. `_normalize_healthkit_point` first reads the metadata field; if absent, falls back to a deterministic hour-of-day heuristic:
  - `05–10` → `breakfast`
  - `11–14` → `lunch`
  - `17–21` → `dinner`
  - else → `snack`
  Wrong sometimes (a 22:00 late dinner becomes `snack`), but the brain also sees the raw timestamp and can override its own interpretation.
- **D-14:** **Output shape parity with Google Fit normalizer**. `_normalize_healthkit_point` emits the same dict shape as `mcp_tools/google_fit_tool._normalize_point`: `{source_id, timestamp, meal_type, calories, protein_g, carbs_g, fat_g, food_item, source}` (with `source="healthkit"`). MealStore consumers (autonomous tick gather, morning briefing aggregate) need no changes.
- **D-15:** **Payload schema = Pydantic `HealthKitPayload` model + locked fixture**. Define `HealthKitPayload` (and a nested `HealthKitSample`) Pydantic model in a new `mcp_tools/healthkit_tool.py`. Required fields: `samples: list[HealthKitSample]`. Each sample requires: `uuid: str`, `start_date: datetime`, `samples_by_type: dict[str, float]` (the dietary HKQuantity values). Optional: `metadata: dict[str, Any]` (where `HKMetadataKeyMealTime` lives), `food_item: str | None`. Lock with `tests/fixtures/healthkit_payload_sample.json` capturing the exact JSON shape the iOS Shortcut emits, and `tests/test_healthkit_payload_schema.py` parsing the fixture into the model + asserting required fields. **Pitfall-6 pattern from Phase 18-04 eval fixtures** — drift on either side breaks the test loudly.

### Multi-source MealStore

- **D-16:** **Google Fit path = keep, mark legacy**. `mcp_tools/google_fit_tool.py`, the `fitness.nutrition.read` OAuth scope, the `fetch_recent_meals` worker-delegated tool registration, and the autonomous-tick `sync_recent_meals` call all stay in place. Add a docstring note on the top of `google_fit_tool.py`: "Legacy Android-source path retained for portability. iOS clients use the HealthKit bridge — see `mcp_tools/healthkit_tool.py`." MealStore stays multi-source-aware. Zero deletion risk.
- **D-17:** **`MealStore` schema is unchanged**. The class is already source-agnostic; the `source` field on each doc just gains a new valid value `"healthkit"`. No schema migration required. No data-retention policy added in this phase (out of scope).

### Observability & operator tooling

- **D-18:** **Heartbeat staleness threshold = 48 hours**. Register `'healthkit-sync': 48` in `_CRON_MAX_STALENESS_HOURS` (`core/heartbeat.py`). Hourly heartbeat alarm fires (via `SelfStateStore` + `update_self_state(stale_cron=...)`) if no push has been recorded in 48h. More sensitive than my standard 7-day suggestion — accepted trade-off: occasional false-alarms on legitimate fast/travel days in exchange for fast detection of a silently-broken bridge.
- **D-19:** **`_log_cron_run('healthkit-sync', ok=True/False)`** at the top of the handler success/exception paths, identical to every other `/cron/*` route. This is what the heartbeat reads to compute staleness.
- **D-20:** **Operator validation tool = `scripts/test_healthkit_push.py` CLI**. From the operator's Mac, builds a realistic-but-fake HealthKit JSON payload (with `source_id="healthkit:test-{timestamp}"` so the test docs are easy to spot and delete after), POSTs to the deployed `/cron/healthkit-sync` using the secret, then asserts the doc lands at `meals/{YYYY-MM-DD}/timestamps/healthkit:test-{ts}` in Firestore. Mirrors `scripts/probe_garmin_export_keys.py` + `scripts/smoke_test_*.py` patterns. End with a "delete test docs?" prompt to clean up.

### SELF.md / brain self-knowledge

- **D-21:** **SELF.md gains a new "Push endpoints" section** generated by `core/self_manifest.py`. Section lists each push-driven endpoint with columns: Endpoint | Driver | Auth | Purpose. First (and only, for now) entry: `/cron/healthkit-sync | iPhone Shortcut | shared-secret bearer | Lifesum nutrition bridge`. Regenerated on every deploy via the existing self_manifest pipeline. Lets the brain truthfully answer "how do my meals reach me?" via SELF.md context, not by spelunking source.

### Eval coverage

- **D-22:** **No new eval fixture**. Existing 5 fixtures (0001-overdue-task ... 0005-calendar-gap) are source-agnostic: the triage prompt sees `meals_since_last_tick` as opaque dicts. Adding a 6th HealthKit-flavoured fixture would cost maintenance without exercising new behaviour. The `_REQUIRED_SNAPSHOT_KEYS` lock from Phase 18-04 (12 keys) stays untouched.

### Documentation & reproducibility

- **D-23:** **`docs/healthkit_shortcut.md` operator runbook + iCloud Shortcut share link**. Step-by-step build instructions for both automations (Lifesum-close 2h-window + 23:55 24h-window catch-up), including: required HealthKit permissions, the exact `Find Health Samples` filter, the `Dictionary` action shape that builds the JSON, the `Get Contents of URL` configuration (method=POST, Authorization header), and the error-branch notification. Plus an iCloud Shortcut share link so the bridge can be rebuilt on a new device in ~5 minutes. Matches `docs/DEPLOYMENT.md` operator-runbook discipline. **Note:** the iCloud link is generated by the operator (Amit) once the Shortcut is built on-device — the plan should leave a placeholder pointer for this.
- **D-24:** **`docs/DEPLOYMENT.md` gains §23 "Push-driven endpoints"** table. Columns: Endpoint | Driver | Auth | Phase. Keeps §19 "Cloud Scheduler Full Job Inventory" cron-only and clean. Future push endpoints (e.g., a hypothetical Apple Watch workout push or a webhook from another service) slot into §23 cleanly. Also adds §24 "HEALTHKIT_WEBHOOK_TOKEN Secret" section mirroring the §20 "TICK_BRAIN_API_KEY (Groq) Secret" pattern: secret name, `--set-secrets` Cloud Run binding, 4-step rotation procedure.

<specifics>
## Specific Ideas

### iOS Shortcut concrete shape (informs but doesn't constrain implementation)

The Lifesum-close Shortcut, written in iOS Shortcuts native actions:

1. **Find Health Samples** — Type: Dietary Energy / Date: "Started in the last 2 Hours" → variable `Energy`
2. **Find Health Samples** — Type: Dietary Protein / Date: same → `Protein`
3. **Find Health Samples** — Type: Dietary Carbohydrates → `Carbs`
4. **Find Health Samples** — Type: Dietary Fat Total → `Fat`
5. **Repeat with Each** sample in `Energy` → build a dict `{uuid, start_date, samples_by_type: {...}, metadata: {...}}` per sample (joining the other macro queries by timestamp / source UUID)
6. **Combine** into a single list `samples`
7. **Dictionary** action → `{"samples": [...]}`
8. **Get Contents of URL** → POST to `https://klaus-agent-XXX.run.app/cron/healthkit-sync`, Headers: `Authorization: Bearer <secret>`, `Content-Type: application/json`, Body: the dictionary as JSON
9. **If** (response status) ≠ 200 → **Show Notification** "Klaus meal push failed (status: <code>)"

This is the operator runbook's foundation — final wording lives in `docs/healthkit_shortcut.md`.

### Pydantic model (sketch, not contract)

```python
class HealthKitSample(BaseModel):
    uuid: str  # HKObject.UUID — stable per-sample identifier
    start_date: datetime
    samples_by_type: dict[str, float]  # e.g. {"DietaryEnergyConsumed_kcal": 547, "DietaryProtein_g": 35.2, ...}
    metadata: dict[str, Any] = {}      # may carry HKMetadataKeyMealTime
    food_item: str | None = None       # may be present if Lifesum wrote a description

class HealthKitPayload(BaseModel):
    samples: list[HealthKitSample]
```

The Shortcut's emitted JSON key names should match — payload schema lock test enforces this.

### `_normalize_healthkit_sample` output shape (must match google_fit_tool)

```python
{
    "source_id": f"healthkit:{sample.uuid}",
    "timestamp": sample.start_date.isoformat(),
    "meal_type": _infer_meal_type(sample),  # HKMetadataKeyMealTime → str, else hour-bucket
    "calories": sample.samples_by_type.get("DietaryEnergyConsumed_kcal", 0),
    "protein_g": sample.samples_by_type.get("DietaryProtein_g", 0),
    "carbs_g":   sample.samples_by_type.get("DietaryCarbohydrates_g", 0),
    "fat_g":     sample.samples_by_type.get("DietaryFatTotal_g", 0),
    "food_item": sample.food_item,
    "source":    "healthkit",
}
```
