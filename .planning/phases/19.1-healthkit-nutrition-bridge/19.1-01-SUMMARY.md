## Key fixture findings (informs Plan 02)

1. **`uuid` arrived empty string.** The `Get Details of Health Samples → Source` action used as a UUID substitute returned `""` rather than an app bundle ID. **Implication for Plan 02:** `HealthKitPayload.uuid: str` accepts empty strings, but the `source_id = f"healthkit:{uuid}"` namespace collapses to `"healthkit:"` for all empty-UUID samples — dedup breaks in production. **Action:** Plan 02 should either (a) generate a deterministic fallback id like `f"healthkit:{start_date}:{calories}"` when `uuid == ""`, or (b) require a non-empty uuid in the Pydantic model (422 on push). Recommend (a) for resilience — production Shortcut may not always expose UUID cleanly. Document as A4 resolution in Plan 02 SUMMARY.
2. **`start_date` arrived as `"2026-05-29T13:00:00GMT+3"`** (non-ISO-8601 timezone suffix). **Implication for Plan 02:** Pydantic V2's default datetime parser may reject this. **Action:** Plan 02 must add a `@field_validator("start_date", mode="before")` that normalizes `GMT+N` → `+0N:00` (or accept the `dateutil` parser as fallback). Add a test covering both `+03:00` and `GMT+3` literal inputs.
3. **`samples_by_type` values arrived as integers (`0`), not strings.** The captured meal didn't have Lifesum nutrition data synced to HealthKit yet, so all macro values returned 0 from the empty `Find Health Samples` queries. **Implication for Plan 02:** the field_validator(mode='before') for string→float coercion (Q1 finding) is still needed for production safety, but cannot be unit-tested off this fixture alone. **Action:** Plan 02's `test_string_numerics_coerce` should use a hand-built dict, not the fixture, to exercise the coercion path.
4. **`metadata` not present.** Lifesum did not write `HKMetadataKeyMealTime` to HealthKit for this sample. **Implication for Plan 02:** the hour-bucket fallback (D-13) is the primary code path; metadata-override path is best-effort. Matches RESEARCH.md Q2 finding.
5. **`DietaryFiber_g` was captured** in addition to the 4 required macros. The normalizer should silently ignore unknown `samples_by_type` keys (via `.get(..., 0)` — already specified in CONTEXT.md `<specifics>`).
