## Fixed Issues

### CR-01: source_id fallback ignores `food_item`, silently dropping distinct meals

**Files modified:** `mcp_tools/healthkit_tool.py`, `docs/healthkit_shortcut.md`, `tests/test_healthkit_tool.py`
**Commit:** 60de6f2
**Applied fix:** Included `food_item` in the Wave-0 synthetic source_id so it
matches the aggregator's `(start_date, food_item)` grouping key. The fallback is
now `healthkit:{start_date_iso}:{food_slug}:{calories_int}` (with `:` escaped to
`_` in the slug). Updated the module docstring, `_compute_source_id` docstring,
and `docs/healthkit_shortcut.md` line 132. Updated
`test_source_id_falls_back_when_uuid_is_lifesum` and added a regression test
(`test_source_id_fallback_includes_food_item_to_avoid_collision`) proving two
foods at the same timestamp with equal integer calories and `uuid="Lifesum"`
now yield two distinct source_ids.

**Logic note:** This is a data-correctness fix; the regression test asserts the
collision is gone, but the developer should confirm the intended key shape on a
real Lifesum push during verification.

### WR-01: Untrusted `uuid` used verbatim as a Firestore document ID

**Files modified:** `mcp_tools/healthkit_tool.py`, `tests/test_healthkit_tool.py`
**Commit:** 3477131
**Applied fix:** The primary-path key now builds from the stripped, sanitized
uuid (`safe = uuid.replace("/", "_")[:200]`) instead of the raw POST-body value,
closing both the `/`-in-document-path hole and the `"ABC "` vs `"ABC"`
idempotency gap. Added `Field(max_length=1024)` on `HealthKitQuantitySample.uuid`
as defense-in-depth against an over-long Firestore document ID. Added three
regression tests: slash sanitization, whitespace-strip idempotency, and
model-level over-length rejection.

Note: the primary-path `safe = uuid.replace("/", "_")[:200]` line lives in
`_compute_source_id` and was introduced in the CR-01 commit (60de6f2) because the
two findings touch the same function; the model-level `max_length` constraint and
the WR-01 regression tests are in this commit.

### WR-02: meal-type inference and source_id fallback assume an aware datetime

**Files modified:** `mcp_tools/healthkit_tool.py`, `tests/test_healthkit_tool.py`
**Commit:** ec3882b
**Applied fix:** Added an `after`-mode `field_validator` (`_ensure_aware`) on
`start_date` that attaches `Asia/Jerusalem` to any naive datetime at parse time,
so `_infer_meal_type`'s `astimezone(_TZ)` no longer assumes UTC on Cloud Run and
the synthetic source_id stays stable across naive/aware variants of the same
instant. Added two regression tests: naive ISO → Jerusalem-aware, and
naive-vs-aware same-instant → identical source_id (idempotency between the
2h-close and 23:55 catch-up pushes).
