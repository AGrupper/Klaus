## Summary

This is a thin plumbing phase, not a research-heavy one. The CONTEXT.md author already did most of the architectural work; the planner's risk surface is concentrated in three places:

1. **iOS Shortcuts wire format is not what the CONTEXT.md Pydantic sketch implies.** Find Health Samples + Repeat With Each + Dictionary actions produce JSON, but each per-sample value can come through as a **string** instead of a number, and "joining" four parallel `Find Health Samples` queries (one per nutrient) by UUID is not a primitive — the Shortcut has to build per-sample dicts manually inside Repeat With Each. The Pydantic model must tolerate stringy numerics and missing fields. [CITED: blog.maximeheckel.com — newline-delimited string shape.]

2. **`HKMetadataKeyMealTime` is not a documented Apple constant.** Apple ships `HKMetadataKeyBloodGlucoseMealTime` (Preprandial/Postprandial — for glucose context, not nutrition). Third-party apps invent their own keys (e.g. `HKFoodMeal` in `rn-apple-healthkit-zen` bindings). D-13 cites `HKMetadataKeyMealTime` with `1=breakfast..4=snack` enum values as if standard — those values are plausibly Lifesum's convention but not confirmed. **Recommendation: treat the hour-bucket heuristic as the PRIMARY code path; tolerate any present `mealTime` / `mealName` / `HKFoodMeal` metadata as a best-effort override. Verify on first real payload via the operator tool.** [VERIFIED: developer.apple.com Metadata Keys index — no HKMetadataKeyMealTime documented.]

3. **`_normalize_point` output `meal_type` is an INT, not a string.** CONTEXT.md `<specifics>` sketch implies `_normalize_healthkit_sample` returns `meal_type: "breakfast"` (string). But the live Google Fit normalizer at `mcp_tools/google_fit_tool.py:159,168,176` returns `meal_type` as an **integer** (1/2/3/4/5 — Google Fit's enum, default 1). D-14 says "parity with Google Fit normalizer." **Recommendation: emit int 1..4 from the HealthKit normalizer (matches Google Fit shape), because MealStore is multi-source and downstream consumers must not branch on source.** Surface this for plan-checker review.

Beyond those three, the codebase patterns are clean ground-truth; the planner can copy `_verify_cron_request` → `_verify_healthkit_request`, copy `cron_autonomous_tick` route shape → `cron_healthkit_sync`, register `'healthkit-sync': 48` in the existing `_CRON_MAX_STALENESS_HOURS` dict, and write the new Pydantic + normalizer + handler against the established Pattern-C try/except discipline.

**Primary recommendation:** ship as 4 waves: Wave 0 schema lock fixture (from real device capture, not hand-written) + tests (RED), Wave 1 normalizer + Pydantic model (GREEN), Wave 2 webhook handler + auth + heartbeat staleness key + SELF.md push-endpoints section, Wave 3 operator CLI + docs + ROADMAP/REQUIREMENTS updates + first-deploy verification.

## Open Questions & Pitfalls

### Q1: iOS Shortcuts wire format — what does the actual JSON look like?

**Finding (MEDIUM confidence):** Shortcuts does NOT emit a clean per-sample structured array out of `Find Health Samples`. Each `Find Health Samples` returns a sequence of typed health values; if you concatenate them into a "Dictionary" action and pass to `Get Contents of URL` with body type JSON, the renderer often flattens children to **newline-delimited text strings**. [CITED: Maxime Heckel — "Building a personal health API with Shortcuts and serverless," documented `hr: "86\n127\n124\n..."` shape.]

**Implication for D-15 Pydantic model:** The naive shape `samples: list[HealthKitSample]` where each sample is `{uuid, start_date, samples_by_type: {...}}` is achievable, but only by structuring the Shortcut as:
1. Four parallel `Find Health Samples` queries (Energy/Protein/Carbs/Fat).
2. **`Repeat With Each` over the Energy result.** Inside the loop: build a Dictionary `{uuid: <sample UUID>, start_date: <ISO 8601 string>, samples_by_type: {DietaryEnergyConsumed_kcal: <number>, ...}, metadata: {...}}` for the current sample by indexing the parallel macro queries on start_date or UUID.
3. Use `Add to Variable` to accumulate the per-sample dicts into a list.
4. Wrap as `{samples: <list variable>}` and POST.

**Quirks the planner MUST handle:**
- **Numeric values may arrive as strings.** Shortcuts coerces Health Sample numeric values to text in the JSON body unless explicit `Get Numbers from Input` is used inside the Repeat loop. The Pydantic model should accept `float | str` and coerce via a `@field_validator(mode="before")`, OR rely on Pydantic V2's permissive `float` coercion (accepts string-numerics, rejects junk).
- **Dates serialize as ISO 8601 with timezone offset.** Pydantic's `datetime` parser handles this natively. No special handling.
- **UUID arrives as the uppercase string form** (e.g. `"D5B6F4E2-7A1C-4F35-B2EE-5E0A9C3D2F1A"`). Pydantic `str` is fine; no need for `UUID` type.
- **Metadata is a flat dict if present, missing entirely if not.** Pydantic `metadata: dict[str, Any] = {}` works.

**Action for planner:** Wave 0 fixture (`tests/fixtures/healthkit_payload_sample.json`) MUST be captured from a real iOS Shortcut fire, not hand-written from imagination. The operator runbook (D-23) needs an early "capture the first payload via webhook.site or a temp print-and-200 endpoint, paste into the fixture" step. Otherwise the schema lock locks the wrong shape and the first real push 422s.

[CITED: https://blog.maximeheckel.com/posts/build-personal-health-api-shortcuts-serverless/ — newline-delimited string shape.]
[CITED: https://support.apple.com/guide/shortcuts/intro-to-using-json-apd0f2e057df/ios — Dictionary action and JSON support.]

### Q2: HKMetadataKeyMealTime reality

**Finding (LOW confidence on D-13 as written):** `HKMetadataKeyMealTime` does NOT appear in Apple's documented HealthKit metadata keys index. The closest documented key is `HKMetadataKeyBloodGlucoseMealTime` (an enum `HKBloodGlucoseMealTime` carrying `Preprandial`/`Postprandial`, for glucose context only — not for general nutrition tagging). [VERIFIED: developer.apple.com/documentation/healthkit/metadata-keys; only blood-glucose meal-time key exists in the public API.]

**What apps actually do:** Third-party nutrition trackers using HealthKit either (a) don't write meal-type metadata at all, leaving categorization to the consumer, or (b) use an app-specific custom string key (e.g. `HKFoodMeal` appears in `jgranstrom/rn-apple-healthkit-zen` bindings; values are arbitrary strings like `"Breakfast"`/`"Lunch"`). [CITED: GitHub `jgranstrom/rn-apple-healthkit-zen/RCTAppleHealthKit+Methods_Dietary.m`.]

**Implication for D-13:** The integer enum `1=breakfast..4=snack` cited in CONTEXT.md is plausibly what Lifesum writes (matches the Google Fit convention used in `_normalize_point`), but is `[ASSUMED]` — not verified against a real iOS payload. The normalizer should tolerate:
- Key entirely missing → fall back to hour-bucket.
- Key present as integer → use as-is, map to int 1..4.
- Key present as string (e.g. `"Breakfast"`) → lowercase + map to int 1..4 if known, else fall back to bucket.
- Unknown integer / string → fall back to bucket.

**Action for planner:** Treat hour-bucket as the **primary** code path, not the fallback. Wrap the metadata-key lookup in a tolerant helper `_extract_meal_type_from_metadata(metadata: dict) -> int | None` that returns `None` on any ambiguity, then the normalizer calls hour-bucket. This is robust to D-13's assumption being wrong.

[VERIFIED: developer.apple.com Metadata Keys index — no HKMetadataKeyMealTime.]
[CITED: GitHub `jgranstrom/rn-apple-healthkit-zen` — HKFoodMeal custom convention.]

### Q3: HealthKit dietary sample edit semantics (UUID stability)

**Finding (HIGH confidence):** HealthKit samples are **immutable**. If a user edits a meal in Lifesum, Lifesum cannot mutate the existing `HKObject`; it must either (a) delete the old sample + insert a new one with a new UUID, or (b) save a sample carrying `HKMetadataKeySyncIdentifier` + `HKMetadataKeySyncVersion`, in which case HealthKit replaces the previous object with the same SyncIdentifier (preserving Lifesum's notion of identity but **changing the HKObject UUID**). [VERIFIED: Apple's HKMetadataKeySyncIdentifier doc — "Saving a HKObject with a HKMetadataKeySyncIdentifier will replace an existing HKObject with the same HKMetadataKeySyncIdentifier value if the new HKObject has a greater HKMetadataKeySyncVersion."]

**Implication for D-12 idempotency:**
- **`source_id = "healthkit:{HKObject.UUID}"` is brittle on edits.** If Lifesum uses SyncIdentifier, an edited meal arrives with a fresh UUID; Klaus upserts a NEW MealStore doc; the old doc remains. The user's daily aggregate now counts the meal twice.
- **The double-count risk is real for the 2h-window trigger:** a meal edited within 2 hours of original log time falls in the same window; both versions arrive (in the same payload or two consecutive fires) and both upsert independently.

**Mitigation options (planner picks):**
1. **Accept the risk as low-likelihood.** Meal edits within 2h are rare; the 23:55 daily catch-up wouldn't double up because the old UUID is gone from HealthKit by the time the catch-up fires (the old `HKObject` was deleted). **This is probably acceptable for v0.**
2. **Prefer SyncIdentifier over UUID when present.** Pydantic model exposes `metadata` — the normalizer can check `metadata.get("HKMetadataKeySyncIdentifier")` first, fall back to `uuid`. This makes `source_id = "healthkit:sync:{syncId}"` for edited meals and `"healthkit:uuid:{uuid}"` for non-synced. More robust, more complexity, requires another fixture variant.

**Recommendation:** Option 1 for this phase; document the risk in `docs/healthkit_shortcut.md`. Defer Option 2 to a follow-up if real-world double-counts surface.

[VERIFIED: Apple HealthKit HKMetadataKeySyncIdentifier semantics, confirmed by `Mylittleswift/ios-health-fitness-apps/HKMetadata.h` source.]

### Q4: "Started in the last 2 hours" filter boundary + timezone

**Finding (MEDIUM):** Apple's official docs are vague on inclusivity. The community consensus is that `Started in the last N hours` is computed at action-execution time as `(now - N hours, now]` — left-exclusive, right-inclusive — using the device's local time zone. [CITED: matthewcassinelli.com — Find Health Samples action page does not document boundary inclusivity.]

**HealthKit's own timezone storage:** Each `HKObject` carries a `HKMetadataKeyTimeZone` metadata field that records the device timezone at write time. This field is **often nil** — Apple's auto-Health writes (steps, heart rate) don't set it, and many third-party apps don't either. [VERIFIED: HKMetadata.h header.]

**Implications for Klaus:**
- **Right-inclusive boundary edge case:** A meal logged exactly 2:00:00 ago will be in the next fire's window AND this fire's window. MealStore.upsert dedupes by source_id (D-12 idempotency anchor), so this is safe.
- **Device-local TZ leak:** The `start_date` in the payload arrives as ISO 8601 with the iPhone's local offset (e.g. `2026-05-28T13:42:00+03:00` in Tel Aviv DST). The current `_normalize_point` for Google Fit uses Asia/Jerusalem-bound datetimes throughout. The HealthKit normalizer should NOT re-zone — just preserve the incoming ISO string and let the date-slicing in `MealStore.upsert` (`meal["timestamp"][:10]`) bucket by the iPhone's local calendar date. **If Amit logs a meal while in another timezone, that meal will bucket by the foreign date** — acceptable trade-off, matches phone-locale intuition.

**Action for planner:** Test fixture includes one sample with non-Asia/Jerusalem offset to confirm the normalizer doesn't crash and the daily bucket is sane.

[VERIFIED: HKMetadataKeyTimeZone documented in HKMetadata.h header file.]
[CITED: matthewcassinelli.com — Find Health Samples documentation.]

### Q5: Shared-secret bearer (D-06) hardening

**Finding (HIGH confidence):** Static-token bearers are an accepted-but-degraded pattern. Best practice for personal-device → cloud webhooks (when HMAC is overkill per D-06) breaks down into 4 must-haves:

1. **Token entropy ≥ 32 bytes.** Use `python -c "import secrets; print(secrets.token_urlsafe(32))"` to mint. 256 bits of entropy is the minimum bar.
2. **Constant-time comparison.** Use `hmac.compare_digest(received_token, expected_token)` — NEVER `==`. A naive `==` leaks token bytes via timing side-channel. [VERIFIED: hooklistener.com webhook-security-fundamentals; pentesttesting.com webhook patterns — `hmac.compare_digest` is the canonical Python check.]
3. **TLS-only.** Cloud Run enforces HTTPS by default; nothing to add. iOS `Get Contents of URL` does NOT support cert pinning. [CITED: Apple Developer Forums — Shortcuts trusts system root CAs only; pinning is not configurable.]
4. **Audit trail.** Log every failed auth attempt at WARNING level with `request.client.host` so brute-force attempts are visible in Cloud Logging.

**Out-of-scope for D-06 (already deferred):** IP allowlist (mobile networks shift IPs unpredictably). HMAC payload signing (declined — overkill for personal device). Replay protection (no nonce/timestamp window) — acceptable because the upsert is idempotent on `source_id`, so a replayed payload produces no new state.

**Rate limiting:** Cloud Run does not natively rate-limit. A leaked-token attacker could DOS the endpoint to drive Cloud Run cost. Mitigations available:
- **Cloud Armor rate-limit rule** on the `/cron/healthkit-sync` path (e.g. 10 req/min per IP). Out of scope for plumbing-only this phase, but worth a note in `docs/healthkit_shortcut.md`'s "Security Considerations" section.
- **Reject empty/malformed payloads fast** so Pydantic 422s before any Firestore write.

**Action for planner:** `_verify_healthkit_request` MUST use `hmac.compare_digest`. Add a unit test that asserts the helper uses the constant-time path (mock `hmac.compare_digest` and assert it's called). Log failed auth at WARNING with redacted token bytes (`token[:4] + "..." + token[-4:]`) for traceability.

[VERIFIED: hooklistener.com webhook security guide; pentesttesting.com webhook patterns.]
[VERIFIED: Apple Developer Forums — Shortcuts cert pinning not configurable.]

### Auth helper shape (`interfaces/web_server.py:232` — `_verify_cron_request`)

```python
async def _verify_healthkit_request(request: Request) -> None:
    """Verify a shared-secret bearer token from the iPhone Shortcut.

    Reads HEALTHKIT_WEBHOOK_TOKEN env (sourced from Secret Manager).
    Constant-time compare via hmac.compare_digest to prevent timing leaks.
    """
    if os.getenv("CRON_DEV_BYPASS", "false").lower() == "true":
        logger.info("CRON_DEV_BYPASS=true — skipping HealthKit auth")
        return
    auth_header = request.headers.get("Authorization", "")
    if not auth_header.startswith("Bearer "):
        raise HTTPException(401, detail={"error": "Missing or malformed Authorization header"})
    received = auth_header.removeprefix("Bearer ").strip()
    expected = os.environ.get("HEALTHKIT_WEBHOOK_TOKEN", "")
    if not expected:
        logger.error("HEALTHKIT_WEBHOOK_TOKEN env var not set — refusing all auth")
        raise HTTPException(500, detail={"error": "Server misconfigured"})
    import hmac
    if not hmac.compare_digest(received.encode(), expected.encode()):
        client = request.client.host if request.client else "?"
        logger.warning("healthkit auth failed from %s (token_prefix=%s...)", client, received[:4])
        raise HTTPException(403, detail={"error": "Invalid token"})
```

**Differences from `_verify_cron_request` (line 232):**
- No OIDC JWT verification; static-token compare instead.
- No `CLOUD_RUN_URL` / `CLOUD_SCHEDULER_SA_EMAIL` env vars touched.
- **`hmac.compare_digest` is mandatory** (Q5).
- 401 for missing/malformed Authorization header; 403 for bad token (matches CONTEXT.md unit test discipline).
- Honour `CRON_DEV_BYPASS` to keep local-dev story consistent with all other `/cron/*` routes.

## Security Domain

`security_enforcement` is implicit (not configured `false`). Applicable ASVS categories:

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | Shared-secret bearer (D-06); `hmac.compare_digest` constant-time; ≥32-byte token entropy (Q5) |
| V3 Session Management | no | Stateless webhook; no session |
| V4 Access Control | yes | Single token = single principal (Amit's iPhone); no role hierarchy needed |
| V5 Input Validation | yes | Pydantic model (D-15); FastAPI auto-422 |
| V6 Cryptography | yes | TLS via Cloud Run; no app-layer crypto (HMAC declined per D-06 / deferred) |
| V7 Error Handling | yes | Log auth failures with redacted token prefix; never return token in error body |
| V8 Data Protection | yes | Token in Secret Manager (`klaus-healthkit-webhook-token`); never logged plaintext |

### Known Threat Patterns for FastAPI + iOS Shortcut

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Token leaked via timing attack on naive `==` | Information Disclosure | `hmac.compare_digest` (Q5) |
| Token leaked via log scraping | Information Disclosure | Log only `token[:4] + "..." + token[-4:]` on failure; never log success path |
| Replay attack | Tampering | Acceptable risk — `MealStore.upsert` is idempotent on `source_id`; replays produce no state delta |
| Brute-force token guessing | Spoofing | 32-byte entropy makes infeasible; consider Cloud Armor rate-limit (deferred to runbook note) |
| Token-in-URL via misconfigured Shortcut | Information Disclosure | Documentation: token MUST be in `Authorization` header, never in URL query string (`docs/healthkit_shortcut.md` must say this explicitly) |
| Malicious payload triggering Firestore quota | Denial of Service | Pydantic 422 happens before Firestore write; cap `samples` list length at e.g. 200 in the model |

## Open Questions

1. **Should the iOS Shortcut emit `samples_by_type` as a strictly-typed numeric dict, or as string-encoded numerics?**
   - What we know: Q1 — Shortcuts often emits numerics as strings.
   - What's unclear: Whether Pydantic V2's coercion is sufficient or a `@field_validator(mode="before")` is required.
   - Recommendation: Add a `@field_validator("samples_by_type", mode="before")` that walks the dict and `float()`-coerces; covers both wire formats.

2. **Cap on samples per payload?**
   - What we know: A 2h window has at most ~3 meals in practice; a 24h catch-up has ~5. A payload of 1000 samples would burn Firestore quota.
   - What's unclear: Whether Pydantic should enforce `Field(max_length=200)` or the handler should slice.
   - Recommendation: Enforce in Pydantic model — fail fast with a 422 rather than partially process and quota-burn.

3. **Should we pre-validate the iOS Shortcut is built correctly before the first deploy?**
   - What we know: Q7 — Wave 0 fixture must be captured from a real device.
   - What's unclear: Whether the planner should ship a one-time `/cron/healthkit-debug` endpoint (with stub auth) that echoes the payload back to the operator's Telegram so they can verify shape before locking the fixture.
   - Recommendation: Out of scope for v0 — webhook.site is good enough for one-time capture. Don't ship a debug endpoint.
