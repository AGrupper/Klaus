# Phase 19.1 Plan 05 (Path B): Flat HealthKit Wire Format + Server-Side Aggregator Summary

Redesigned the HealthKit Shortcut → /cron/healthkit-sync contract from bundled per-meal samples to flat per-quantity samples after live UAT proved the bundled shape was structurally unbuildable from iOS Shortcuts.

## What landed

- **New `HealthKitQuantitySample` Pydantic model** in `mcp_tools/healthkit_tool.py` — one HKQuantitySample per row: `{uuid, start_date, quantity_type, value, metadata, food_item}`. Preserves the GMT+N start_date validator and the string-numeric value coercer.
- **Old `HealthKitSample` (bundled) model deleted.** Clean replacement, no shim. Phase had not shipped; keeping both would have just confused the wire format.
- **New `_aggregate_quantity_samples()` function** — groups flat samples by `(start_date.isoformat(), food_item)` and **sums** same-`quantity_type` values within each group (handles Lifesum's per-food-item fan-out: a 3-piece chicken meal writes 3 protein samples that must sum into one).
- **`ingest_payload` updated** to parse flat → aggregate → normalize → upsert. Per-meal try/except (Pattern C) means one malformed group does not drop the batch. Output count = distinct (start_date, food_item) groups, NOT raw sample count.
- **`_normalize_healthkit_sample` now consumes plain dicts** (the aggregator's output) instead of Pydantic instances. The 9-key output shape is unchanged so morning briefing, autonomous tick, and MealStore see identical per-meal records. `_infer_meal_type` and `_compute_source_id` were refactored alongside to read from dicts.
- **`HealthKitPayload.samples` DoS cap bumped 200 → 1000** to accommodate fan-out (e.g. 8 meals/day × 5 macros × 5 food items = 200 with comfortable headroom).
- **Fixture replaced** with the operator's real 2026-05-29 20:00 meal: 324 kcal / 5.4 g protein / 43.2 g carbs / 14.4 g fat / 2.4 g fiber, in flat-row format with the GMT+3 timezone suffix preserved (exercises the date validator).
- **Schema-lock test updated** — `_REQUIRED_SAMPLE_KEYS = {"uuid", "start_date", "quantity_type", "value"}`.
- **8 new aggregator + flat-payload tests** added to `tests/test_healthkit_tool.py` (groups-five-macros, sums-duplicates, separate-by-start-date, separate-by-food-item, real-fixture-values, flat-payload-canonical-parse, string-value-coerces, non-coercible-value-raises). 4 ingest_payload tests reworked for per-meal semantics.
- **`TestCronHealthkitSync` updated** — happy-path and idempotent-repush tests now compute expected upsert count as distinct (start_date, food_item) groups, not raw sample count. The Path-B fixture's 5 samples aggregate to 1 meal → `{"upserted": 1}`.
- **`scripts/test_healthkit_push.py::_build_synthetic_payload` updated** — emits 4 flat samples per synthetic meal (Energy/Protein/Carbs/Fat) keyed by distinct `food_item` per meal so `count` distinct Firestore docs still land (verification path semantics unchanged).
- **`docs/healthkit_shortcut.md` revised** — §1 reworked with the flat wire format + worked example using the operator's real meal; §3 step-by-step build replaced with a Path-B placeholder describing the "dumb flat pipe" design at high level (5 Find Health Samples per macro, Repeat with Each emitting flat dicts, concat + POST); §4 catch-up reference updated; §8 troubleshooting table refreshed.
- **`19.1-CONTEXT.md` D-15 addendum** — documents why the bundled contract was structurally infeasible (HKCorrelation parent UUIDs unreachable from Shortcuts' Find Health Samples action) and what replaced it, with the real-device totals locked as ground truth.

## Why Path B (root cause of the UAT bug)

Lifesum on iPhone writes **one `HKQuantitySample` per macro per food item**. A meal with 3 food items writes 3 Energy + 3 Protein + 3 Carbs + 3 Fat + 3 Fiber samples — 15 separate HK objects.

Apple's HealthKit groups these under an `HKCorrelationTypeIdentifierFood` parent (an `HKCorrelation` whose UUID is the conceptual "meal ID"), but **iOS Shortcuts has no action to read that parent UUID from a `Find Health Samples` result.** The `Get Details of Health Sample` action exposes Source / Start Date / Quantity / Type — but not the correlation parent ID. So a Shortcut iterating over Energy samples cannot say "give me the Protein/Carbs/Fat samples that belong to the same HKCorrelation as this Energy sample" — it can only filter by start_date, which collapses every food item in a multi-item meal into one (or, if the timestamps differ even slightly across food items as Lifesum sometimes does, splits a single meal across multiple groups).

The original bundled `HealthKitSample` shape (`{uuid, start_date, samples_by_type: {kcal, protein, carbs, fat}}`) assumed the client could reassemble per-meal records before POST. It can't. Path B moves that aggregation server-side where dicts and grouping are trivial. The wire format becomes "dumb flat pipe," and the Pydantic + grouping + summing logic on Klaus's side runs ~30 lines of Python.

## Deviations from Plan

- **[Rule 1 fix] `_coerce_stringy_numeric` semantics changed.** The old `samples_by_type` validator silently dropped non-coercible inner-dict entries. In Path B, each sample row carries exactly one `value`, so a garbage value now raises `ValidationError` for that row. The per-meal try/except in `ingest_payload` still prevents the batch from dropping. The test `test_flat_payload_non_coercible_value_raises` locks this behaviour. This is a deliberate semantics shift (cleaner per-row reporting), called out in the model docstring.
- **[Rule 3 fix] Updated `TestCronHealthkitSync::test_happy_path_upserts_each_sample` → `test_happy_path_upserts_each_meal`.** The original test name and arithmetic (`expected_n = len(fixture["samples"])`) presumed one upsert per raw sample. Path B's whole point is that the server collapses N samples into M ≤ N meals, so the expected count is the distinct `(start_date, food_item)` group count. Renamed to match.
