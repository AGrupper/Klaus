# Phase 19.1 Plan 03: Webhook Handler + Bearer Auth Summary

**Substantive one-liner:** POST /cron/healthkit-sync FastAPI route + shared-secret `_verify_healthkit_request` helper using `hmac.compare_digest` constant-time compare; delivers HEALTHKIT-04 + HEALTHKIT-05 — the inbound attack surface of the bridge.

## What landed

- **`interfaces/web_server.py`** grew from 480 → 601 lines:
  - `_verify_healthkit_request(request)` — async, inserted at line 278 right after `_verify_cron_request`. Honours `CRON_DEV_BYPASS=true`. Raises:
    - `HTTPException(401, {"error": "Missing or malformed Authorization header"})` on missing/non-Bearer header.
    - `HTTPException(500, {"error": "Server misconfigured"})` when `HEALTHKIT_WEBHOOK_TOKEN` env var is empty (refuse-all, prevents fail-open per threat T-19.1-03-08).
    - `HTTPException(403, {"error": "Invalid token"})` after `hmac.compare_digest(received.encode(), expected.encode())` returns False. Failure logs at WARNING with `received[:4] + "..." + received[-4:]` redacted prefix + client IP — token is never written to logs in full.
  - `cron_healthkit_sync(request)` @ `POST /cron/healthkit-sync` — inserted right after `cron_autonomous_tick`. Lazy-imports `mcp_tools.healthkit_tool`, `MealStore`, `pydantic.ValidationError`. Flow: verify → `payload_json = await request.json()` → `ingest_payload(payload_json, MealStore(project_id, database))` → return `JSONResponse({"upserted": result["upserted_count"]})`. `ValidationError` is caught mid-handler and re-raised as `HTTPException(422)` with the Pydantic error list. All paths call `_log_cron_run("healthkit-sync", ok=True|False)`; uncaught exceptions are re-raised so Cloud Run sees the 500.

  - `class TestCronHealthkitSync` — 11 tests covering:

    2. `test_bad_token_returns_403`

## Does the compare_digest mock pattern work?

Yes — with one subtlety worth recording. The auth helper does **not** lazy-import `hmac` inside the function body; web_server.py already imports `hmac` at module top (line 23) for the existing `/telegram-webhook` token check. So the patch target is `interfaces.web_server.hmac.compare_digest` — patched via `patch.object(ws.hmac, "compare_digest", digest_mock)` after the module is loaded by the `_ws_module` fixture. The mock's `call_args_list` is iterated to assert each call received `bytes` arguments (proving the canonical `received.encode(), expected.encode()` shape was used).

The plan's note about "lazy `import hmac` inside the helper body" was a hold-over from the PATTERNS skeleton; since the module-level import was already present, no new lazy import was added.

## Was an auth-flow refactor (`_verify_cron_request` + `_verify_healthkit_request` → shared helper) tempted?

Yes — both helpers share the `CRON_DEV_BYPASS` short-circuit and the `Bearer ` header parse. But CONTEXT.md `Claude's Discretion` says **NO**: the two verification models (OIDC JWT for Cloud Scheduler vs. shared-secret bearer for the iPhone) have fundamentally different shapes (token is verified via Google's library vs. directly compared via hmac; failure modes return different status codes; SA-email check vs. token-prefix redaction). Folding them together would obscure the security model and force a parameter-heavy abstraction. Kept them parallel-but-separate. The shared `CRON_DEV_BYPASS` + Bearer-prefix idiom (two lines each) is acceptable duplication.

## CPython 3.14 dotenv/GC segfault — why module-scoped fixture

The autonomous-tick tests' `_stub_web_server_imports()` pattern deletes `interfaces.web_server` from `sys.modules` and re-imports it on each test. With 5 tests this is fine, but at 11 tests in one process the dotenv module's `typing.NamedTuple` machinery + CPython 3.14's GC pass during repeated module destruction triggers a segfault inside `collections.namedtuple` (reproducible via `pytest tests/test_web_server.py::TestCronHealthkitSync` with the per-test reimport pattern — crash after test 9, ~10s in).

**Fix:** Module-scoped `_ws_module` fixture imports web_server **once** per test module (with the stubs applied at fixture setup). Tests then mutate `ws._application` / `ws._log_cron_run` directly and restore in `try/finally`. Per-test `os.environ` patches still work because every test re-enters the `patch.dict(os.environ, env)` context before constructing the TestClient.

This is a per-Python-version environmental quirk, not a code defect. Other test files already use module-scoped imports; the autonomous-tick class's per-test reimport remained on the older 5-import pattern only because it predates 3.14.

- **Fix:** Constructed `MealStore(project_id=os.environ.get("GCP_PROJECT_ID", ""), database=os.environ.get("FIRESTORE_DATABASE", "(default)"))` inside the handler. Mirrors the in-codebase pattern at `core/autonomous.py:321`, `core/morning_briefing.py:250`, and every other Firestore store-construction site.

- **Commit:** `6296f8a` (folded into GREEN commit)
