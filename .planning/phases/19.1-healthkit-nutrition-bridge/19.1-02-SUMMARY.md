# Phase 19.1 Plan 02: HealthKit Normalizer + Pydantic Model Summary

**Substantive one-liner:** Pydantic V2 HealthKit payload model + 9-key meal-dict normalizer with Wave-0 Lifesum source_id fallback and GMT+N start_date coercion.

## What landed

- **`mcp_tools/healthkit_tool.py`** (264 lines, exceeds plan's ≥120 floor):
  - `HealthKitSample` + `HealthKitPayload` Pydantic V2 models
  - Two `@field_validator(mode="before")` decorators:
    - `_normalize_gmt_offset` on `start_date` — rewrites Wave-0 `GMT+3` → `+03:00`
    - `_coerce_stringy_numerics` on `samples_by_type` — `float()`-coerces stringy values; drops non-coercible with `logger.warning`
  - `Field(max_length=200)` DoS guard on `samples`
  - `_hour_bucket(hour)` returns int 1..4 (5-10→1, 11-14→2, 17-21→3, else→4)
  - `_infer_meal_type(sample)` — `HKMetadataKeyMealTime` override (int 1..4 OR string `"Breakfast"`/`"Lunch"`/...) → hour-bucket fallback
  - `_compute_source_id(sample)` — primary `healthkit:{uuid}`; Wave-0 fallback to `healthkit:{start_date_iso}:{calories_int}` when uuid is empty or matches `_KNOWN_SOURCE_NAMES = {"Lifesum", "Apple Health", "Health"}`
  - `_normalize_healthkit_sample(sample)` emits the 9-key dict matching `google_fit_tool._normalize_point`
  - `ingest_payload(payload_dict, store)` — per-sample try/except (Pattern C); returns `{"upserted_count": N, "errored_count": M}`
  - `HealthKitUnavailableError` custom exception (mirrors `GoogleFitUnavailableError`)

## Pydantic V2 `field_validator(mode="before")` behaviour confirmed

Confirmed against the real-device fixture: the `_normalize_gmt_offset` validator successfully rewrote `2026-05-29T07:30:00GMT+3` to `2026-05-29T07:30:00+03:00` before Pydantic's default datetime parser ran, producing a tz-aware `datetime` with `+03:00` offset. Cross-checked with the equivalent ISO-8601 literal `2026-05-29T07:30:00+03:00` — both parse to the same instant (test `test_start_date_gmt_plus_n_parses` enforces this regression-proof).

Stringy-numeric coercion verified against a synthetic mock dict (the real Wave-0 fixture happened to arrive with numeric `int`/`float` types — Wave 0 finding #3 — so the coercion path is exercised only by the mock-dict test, which is sufficient and exactly what the Wave 0 summary called for).

## Did the real-device fixture expose Lifesum's actual metadata key?

**No.** Wave 0 confirmed `metadata` is not present in Lifesum-written samples (finding #4). RESEARCH.md Assumption **A1** ("`HKMetadataKeyMealTime` is Lifesum's chosen key name") remains UNVERIFIED — the hour-bucket fallback is the primary production code path. The metadata-override path is implemented and unit-tested (`test_metadata_override_takes_precedence`, `test_metadata_string_breakfast_maps_to_one`) but will not fire in production until Lifesum (or a future first-party Klaus writer) populates the metadata field. No assumption update is required for RESEARCH.md.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Critical functionality] Wave 0 Lifesum source_id fallback**

- **Found during:** Task 2 design (called out explicitly in `<critical_context_from_wave_0>` and `19.1-01-SUMMARY.md` finding #1)
- **Issue:** The plan's `f"healthkit:{sample.uuid}"` source_id literal would collide for every Lifesum-written sample (all collapse to `healthkit:Lifesum`), breaking dedup in production.
- **Fix:** Added `_compute_source_id(sample)` that returns the primary `f"healthkit:{sample.uuid}"` literal (preserves the plan's grep-anchor) but falls back to a deterministic `f"healthkit:{start_date_iso}:{calories_int}"` when uuid is empty or in `_KNOWN_SOURCE_NAMES = {"Lifesum", "Apple Health", "Health"}`. Two tests cover the fallback path (`test_source_id_falls_back_when_uuid_is_lifesum`, `test_source_id_falls_back_when_uuid_empty`).
- **Files modified:** `mcp_tools/healthkit_tool.py`, `tests/test_healthkit_tool.py`
- **Commits:** `ddce545`, `569d6f0`

**2. [Rule 2 — Critical functionality] GMT+N start_date Pydantic coercion**

- **Found during:** Task 2 (called out in `<critical_context_from_wave_0>` finding #2)
- **Issue:** The Wave-0 fixture's `2026-05-29T07:30:00GMT+3` is not ISO 8601 and Pydantic V2's default datetime parser rejects it (verified empirically: `ValidationError: unexpected extra characters`).
- **Fix:** Added `@field_validator("start_date", mode="before")` (`_normalize_gmt_offset`) that uses a compiled regex `GMT([+-])(\d{1,2})(?::?(\d{2}))?$` to rewrite the suffix to `+0N:00` before Pydantic's parser runs. Test `test_start_date_gmt_plus_n_parses` asserts both `GMT+3` and `+03:00` literals parse to the same datetime.
- **Files modified:** `mcp_tools/healthkit_tool.py`, `tests/test_healthkit_tool.py`
- **Commits:** `ddce545`, `569d6f0`

**3. [Rule 2 — Defensive default] `samples_by_type.get(..., 0)` instead of `0.0`**

- **Found during:** Task 2 implementation review
- **Issue:** Plan code-sketch used `0.0` defaults; the test `test_normalize_missing_macro_keys_defaults_to_zero` asserts equality with `0` (int) which is duck-type compatible. Switched to integer `0` defaults to match the lifelong-zero downstream consumer pattern (MealStore aggregates), preserving int/float interoperability.
- **Files modified:** `mcp_tools/healthkit_tool.py`
- **Commits:** `569d6f0`
