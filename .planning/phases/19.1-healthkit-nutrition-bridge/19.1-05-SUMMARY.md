## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] MealStore construction signature**

- **Found during:** Task 1 implementation review (cross-referencing Plan 03's auto-fix note for the same bug).
- **Issue:** Plan's literal code-sketch in `<action>` for `_verify_firestore` used `MealStore()` zero-arg. The actual signature in `memory/firestore_db.py` is `__init__(self, project_id: str, database: str = "(default)")` — same trap Plan 03 fell into. If left unfixed, the smoke CLI would crash with `TypeError: missing 1 required positional argument: 'project_id'` when the operator fires it.
- **Fix:** Construct `MealStore(project_id=os.environ.get("GCP_PROJECT_ID", "klaus-agent"), database=os.environ.get("FIRESTORE_DATABASE", "klaus-firestore"))`. Defaults are the live infra values (CLAUDE.md §5) so the operator-Mac flow works out of the box if env vars are unset (with the documented `export GCP_PROJECT_ID=klaus-agent` opt-in for clarity).
- **Files modified:** `scripts/test_healthkit_push.py` (initial write; no separate commit needed).
- **Commit:** `79b8bb7` (folded into the GREEN commit).

**2. [Rule 3 — Blocking] Firestore client kwargs for cleanup path**

- **Found during:** Task 1 implementation review.
- **Issue:** Plan's literal code-sketch in `<action>` for the cleanup `firestore.Client(database="klaus-firestore")` omits `project=...`. Live Firestore SDK accepts `project` and falls back to `GOOGLE_CLOUD_PROJECT`, but the operator may not have that env set when running the smoke CLI on their Mac. Without explicit `project=`, the cleanup `.delete()` may fail silently or hit the wrong project.
- **Fix:** Construct `firestore.Client(project=project_id, database=database)` with the same env-var resolution as the MealStore path (preserving cleanup idempotency).
- **Files modified:** `scripts/test_healthkit_push.py`.
- **Commit:** `79b8bb7` (folded into the GREEN commit).

**3. [Rule 2 — Defensive default] CLI `--token-env` indirection retained**

- **Found during:** Task 1 implementation review.
- **Issue/decision:** The plan code-sketch's `--token-env` flag (default `HEALTHKIT_WEBHOOK_TOKEN`) is preserved. Rationale: lets operator pass a different env var name during rotation testing without re-exporting (e.g., `HEALTHKIT_WEBHOOK_TOKEN_NEW`). Trivial code addition (~3 lines); zero risk; matches the plan exactly.

### Task 3 commands (operator runs from Mac terminal)

```bash
# 1. Mint a 32-byte token
NEW_TOKEN=$(python -c "import secrets; print(secrets.token_urlsafe(32))")
echo "$NEW_TOKEN" > /tmp/klaus-healthkit-token  # for pasting into Shortcut later

# 2. Create the Secret Manager secret + add the first version
gcloud secrets create klaus-healthkit-webhook-token \
    --replication-policy=automatic \
    --project=klaus-agent 2>/dev/null || true
printf '%s' "$NEW_TOKEN" | gcloud secrets versions add \
    klaus-healthkit-webhook-token --data-file=- --project=klaus-agent

# 3. Bind to Cloud Run + redeploy
gcloud run services update klaus-agent \
    --region=me-west1 \
    --update-secrets=HEALTHKIT_WEBHOOK_TOKEN=klaus-healthkit-webhook-token:latest \
    --project=klaus-agent

# 4. Wait ~30s for the new revision to take traffic, then fire the smoke test
export HEALTHKIT_WEBHOOK_TOKEN="$NEW_TOKEN"
export GCP_PROJECT_ID=klaus-agent
python scripts/test_healthkit_push.py \
    --url https://klaus-agent-XXXX.run.app/cron/healthkit-sync \
    --count 2
```
