expected: After ingest, asking Klaus reflects the logged calories/macros — Klaus reads MealStore (where HealthKit writes), not the dead Google Fit source.
result: issue
reported: "Asked Klaus 'What did I eat today?' (2026-05-30 16:07). He replied: 'there are no nutrition entries recorded in Google Fit or Lifesum for today ... either nothing has been logged, or the sync from Lifesum has not yet completed.' Klaus is querying GOOGLE FIT, not MealStore — so HealthKit-ingested meals are invisible to the on-demand query."
severity: major
analysis: |
  Root cause (confirmed in code before the tool environment began returning
  corrupted output):
  - The brain-direct `fetch_recent_meals` tool handler routes to
    `core/tools.py::_handle_fetch_recent_meals` → `get_google_fit().fetch_recent_meals(hours=...)`
    — i.e. it reads the DEPRECATED Google Fit source, which returns [] on iOS.
    It does NOT read `MealStore` (where `/cron/healthkit-sync` writes HealthKit meals).
  - This is the SAME class of defect as the autonomous tick
    (`core/autonomous.py:319` `sync_recent_meals()` from google_fit) — the meal
    READ paths were never repointed from Google Fit to MealStore when the iOS
    HealthKit bridge replaced the (dead-on-iOS) Google Fit ingest.
  - Klaus's own wording ("Google Fit or Lifesum") is the smoking gun: the tool
    he called names Google Fit.
  Scope correction: the iOS-source fix is broader than originally scoped for
  Phase 19.3. It must cover BOTH read paths:
    (a) the autonomous-tick gather (`core/autonomous.py:319`), AND
    (b) the brain-direct `fetch_recent_meals` tool (`core/tools.py` ~line 139-142).
  Morning briefing (`core/morning_briefing.py:254`) is the only path that already
  reads MealStore correctly.
  NOTE: this is NOT a Phase 19.1 regression — 19.1 correctly delivered the WRITE
  path (HealthKit → MealStore). The READ paths are Phase 19 carryover wired to
  Google Fit. Captured as Phase 19.3 (scope broadened below).
