## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cron/brain → Firestore `coaching_topics` | Internally-derived topic keys written; no user-supplied key crosses here |
| Telegram (allowlisted user) → training_log notes | Notes text reaches derive_session_quality keyword scan |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-01 | Tampering | CoachingTopicStore.add_topic topic_key | mitigate | topic_key values are always internally derived from a fixed `category:subject` vocabulary (Finding 2 table) — never sourced from user input; no path-concat, value stored as a plain string in ArrayUnion |
| T-24-02 | Tampering | derive_session_quality notes keyword scan | accept | notes originate only from the allowlisted Telegram user (TELEGRAM_ALLOWED_USER_IDS); override keywords are hardcoded constants, not user-supplied patterns; worst case is a self-inflicted mislabel of own session quality |
| T-24-03 | Denial of Service | CoachingTopicStore reads inside crons | mitigate | has_topic / topics_today are fail-open (return False / []) and never raise, so a Firestore read error cannot crash a cron (data-integrity bias: prefer firing a topic over silent suppression) |
| T-24-04 | Information Disclosure | Firestore writes (PII in logs) | mitigate | error logs use `exc_info=True` + the topic key / date only — no meal contents, no biometrics; quality is a 3-value enum, not raw health data |
| T-24-SC | Tampering | npm/pip/cargo installs | N/A | no new packages introduced in this plan (RESEARCH Package Legitimacy Audit: not applicable) |
