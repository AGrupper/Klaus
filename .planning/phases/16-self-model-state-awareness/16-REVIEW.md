### CR-01: Plans 16-02 and 16-03 never merged into main — core Phase 16 functionality absent

**Files:** `core/main.py`, `core/tools.py`, `prompts/smart_agent.md`
**Issue:** The three commits that implement the core Phase 16 runtime behavior are on dead worktree branches (`worktree-agent-ae960d9eab373431e`) and are not reachable from `main`:

- `2780453` — `AgentOrchestrator.__init__` SELF.md loading + `SelfStateStore` bootstrap

### CR-02: CI workflow runs `python core/self_manifest.py` without installing Python dependencies

**File:** `.github/workflows/deploy.yml:47`
**Issue:** The "Generate SELF.md capability manifest" step runs `python core/self_manifest.py` immediately after checkout and GCloud auth, with no preceding `pip install` step. `core/self_manifest.py` imports from `dotenv` and conditionally from `googleapiclient`, `google.cloud.firestore`, and `google.oauth2`. Although `_load_tool_data` stubs these when they are missing, `dotenv` is imported unconditionally at the top of `memory/firestore_db.py` and `core/main.py`. If the runner's Python environment does not have `python-dotenv` pre-installed, the step will fail with `ModuleNotFoundError` before any stub logic runs.

The `Generate SELF.md` step must not be the root cause of a failed deploy.

**Fix:**
```yaml
- name: Install manifest dependencies
  run: pip install python-dotenv --quiet

- name: Generate SELF.md capability manifest
  run: python core/self_manifest.py
```
Alternatively, if a `requirements.txt` is present, install it in full before the manifest step — that matches what the Docker build does and removes the discrepancy entirely.

## Warnings

### WR-01: `asyncio.get_event_loop().run_until_complete()` is deprecated and unsafe in async context

**File:** `core/heartbeat.py:673`
**Issue:** `_drain_quiet_queue` calls `asyncio.get_event_loop().run_until_complete(send_and_inject(...))`. This is called from `run_tick()`, which is an `async def` function. Calling `run_until_complete()` from inside a running event loop raises `RuntimeError: This event loop is already running` on Python 3.10+ (the default Cloud Run Python runtime). The drain will always silently fail (swallowed by the outer `except Exception`) whenever it is invoked in the normal async path, meaning queued signals accumulated during quiet hours are never drained.

**Fix:** Replace with `await`:
```python
async def _drain_quiet_queue(bot, now: datetime, config: dict) -> None:
    if _in_quiet_hours(config, now):
        return
    ...
    signals = [Signal(**s) for s in queued]
    await send_and_inject(bot, _compose_message(signals), inject_into_conversation=True)
    doc_ref.delete()
```
And update the call site in `run_tick` to `await _drain_quiet_queue(bot, now, config)`.

### WR-03: `core/main.py` calls `load_dotenv()` without `override=True`

**File:** `core/main.py:36`
**Issue:** The project memory explicitly documents that `load_dotenv()` must always be called with `override=True` (see `feedback_dotenv_override.md`). The call at line 36 uses the default, which silently ignores `.env` values when the shell has already exported the variable. This is a known footgun in this project: local development against a `.env` file that has different values than the shell environment will appear to work but will silently use the stale shell-exported values.

**Fix:**
```python
load_dotenv(override=True)
```

### IN-02: SHA-1 hash algorithm used in `self_manifest.py` and `heartbeat.py`

**Files:** `core/self_manifest.py:78`, `core/heartbeat.py:500`
**Issue:** Both files use `hashlib.sha1()` for the schema-staleness fingerprint. SHA-1 is not being used in a security context here (it is a content-change detector, not a cryptographic commitment), so this is not a vulnerability. However, Python 3.9+ emits a `DeprecationWarning` for bare `hashlib.sha1()` calls in FIPS-enabled environments. To future-proof, pass `usedforsecurity=False`.

**Fix:**
```python
return hashlib.sha1(combined.encode(), usedforsecurity=False).hexdigest()
```
Apply in both `_compute_schema_hash()` in `self_manifest.py` and the inline hash in `check_code()` in `heartbeat.py`.

### IN-03: `_load_tool_data` in `self_manifest.py` does not clean up successfully-loaded stubs

**File:** `core/self_manifest.py:163-170`
**Issue:** `_ensure_stub()` registers stub modules into `sys.modules` and records them in `_stubs`. On failure, the `except` block at line 168 pops them (`sys.modules.pop(mod_name, None)`). However, on **success** (when `exec_module` completes without raising), the stubs are never removed from `sys.modules`. If `generate_manifest()` is called more than once in the same process (e.g., in tests or a long-running process), the stubs remain and may interfere with subsequent real imports of `google.cloud.firestore`, `googleapiclient`, etc.

**Fix:** Add a cleanup block after successful execution:
```python
spec.loader.exec_module(module)
# Clean up stubs regardless of success — don't pollute sys.modules
for mod_name in _stubs:
    sys.modules.pop(mod_name, None)
```
