# Phase 30.5 Plan 01: Tick-Brain Fallback Decoupling Summary

**TickBrain's Groq-failure fallback now reads TICK_BRAIN_FALLBACK_BACKEND/MODEL/API_KEY/BASE_URL (default gemini/gemini-3.5-flash) instead of the smart brain's SMART_AGENT_* vars, wired into deploy.yml and unit-proven to never route to a Claude model — the milestone's central cost-safety fix (BRAIN-03) ahead of the Plan 06 brain flip.**

## Performance

- **Duration:** ~20 min
- **Completed:** 2026-07-17T19:26:46Z
- **Tasks:** 2 of 3 completed (Task 3 is a blocking live-verify checkpoint — see below)
- **Files modified:** 3

## Accomplishments
- `TickBrain.__init__` no longer reads any `SMART_AGENT_*` var; fallback client construction reads `TICK_BRAIN_FALLBACK_BACKEND`/`_MODEL`/`_API_KEY`/`_BASE_URL`, defaulting backend/model to `gemini`/`gemini-3.5-flash` when unset, but only constructing the client when an API key is present
- `tests/test_tick_brain.py` rewritten/extended: new constructor tests assert the fallback is built from `TICK_BRAIN_FALLBACK_*` env, defaults apply when backend/model are unset, the fallback stays `None` without an API key, the module source never contains the string `SMART_AGENT`, and — following the RESEARCH.md Code Examples shape verbatim — a forced primary (Groq) `LLMError` with `SMART_AGENT_MODEL` already flipped to `claude-sonnet-5` in the environment still routes to `_fallback_model == "gemini-3.5-flash"`, never the Claude model
- `.github/workflows/deploy.yml` carries `TICK_BRAIN_FALLBACK_BACKEND=gemini,TICK_BRAIN_FALLBACK_MODEL=gemini-3.5-flash` in `--set-env-vars` and `TICK_BRAIN_FALLBACK_API_KEY=klaus-gemini-key:latest` in `--update-secrets`; `SMART_AGENT_MODEL=claude-sonnet-5` is confirmed absent (the flip is Plan 06's job, not this plan's)

## Decisions Made
- Fallback client construction gated strictly on API-key presence (not merely backend/model defaults) — matches the plan's explicit "preserve the existing optional-client shape" instruction and avoids constructing a client with no usable credential.
- Reused the existing `klaus-gemini-key` secret for `TICK_BRAIN_FALLBACK_API_KEY` rather than provisioning a new one, per RESEARCH.md Assumption A3 (same key already backs `SMART_AGENT_API_KEY` today).
- Placed the `TICK_BRAIN_FALLBACK_*` env values immediately adjacent to the existing `TICK_BRAIN_*` block in `deploy.yml` for readability, and the secret mapping immediately after `SMART_AGENT_FALLBACK_API_KEY` for the same reason.
