## Summary

This phase swaps Klaus's smart brain from `gemini-3.5-flash` to `claude-sonnet-5`, activates
Anthropic prompt caching with an explicit 1-hour TTL, extracts and correctly prices
cache-read/cache-write tokens, decouples the tick-brain's Groq-failure fallback from the
brain's env vars (must ship *before* the model flip), adds a once-daily heartbeat spend
tripwire, does a light de-prescription pass on the always-on prompt plus a compact SELF.md
generator rewrite, and TTL-caches `UserProfileStore` reads. There are no new libraries, no new
datastores, and no new hosting primitives — this is a floor-bump on the already-integrated
`anthropic` SDK plus disciplined changes to three files that already exist:
`core/llm_client.py`, `core/pricing.py`, `memory/firestore_db.py`.

A milestone-wide research pass (`.planning/research/{SUMMARY,STACK,PITFALLS,ARCHITECTURE}.md`,
same date as this research) already investigated this phase in depth against official Anthropic
docs and direct codebase inspection. This phase-level research reuses those verified findings
(re-confirmed against the live code during this session — file:line citations below) and adds
the specific implementation detail the planner needs: current call-site behavior, the exact
diff shape for `_AnthropicBackend`, the exact `UserProfileStore`/`SelfStateStore` cache-pattern
delta, and a requirement-by-requirement build sequence.

**Primary recommendation:** Follow the internal build order already locked in
`.planning/research/ARCHITECTURE.md` §"Suggested Build Order, step 2" — decouple
`TICK_BRAIN_FALLBACK_*` first, then the Anthropic compatibility + cache-token audit (same
change as the cost tripwire dependency), then pricing/LLMUsage fields, then flip
`SMART_AGENT_*`, then slim the prompt, then wire the heartbeat tripwire, with the
`UserProfileStore` TTL cache landing any time (it's independent). Do not reorder — every step
after the first two depends on truthful metering existing before the tripwire consumes it.

### Pattern 1: Anthropic system-prompt cache_control block

**What:** `system` must become a content-block list carrying `cache_control` on the last
(most-inclusive) block, rather than a plain string.
**When to use:** Every Anthropic-backend call in `_AnthropicBackend.chat()` when `system` is
non-empty — this is the single change that activates caching.
**Example:**
```python
# Source: platform.claude.com/docs/en/build-with-claude/prompt-caching (via .planning/research/STACK.md)
# Today (core/llm_client.py:210-211):
if system:
    kwargs["system"] = system

# Target shape:
if system:
    kwargs["system"] = [
        {
            "type": "text",
            "text": system,
            "cache_control": {"type": "ephemeral", "ttl": "1h"},
        }
    ]
```
Tool schemas can carry their own `cache_control` on the last tool definition to cache the tool
block as a separate breakpoint (max 4 breakpoints per request — a 3-breakpoint layout of
system / tools / conversation-tail leaves one spare, per `.planning/research/STACK.md`).

### Pattern 2: Cache-token usage extraction + pricing

**What:** Extract `cache_creation_input_tokens`/`cache_read_input_tokens` from
`response.usage` (today only `input_tokens`/`output_tokens` are read) and thread them through
`compute_cost` and `LLMUsageStore.record`.
**When to use:** Same change as Pattern 1 — never ship `cache_control` without this, per
Pitfall 3 below.
**Example:**
```python
# Source: verified against core/llm_client.py:240-248 (current code) +
# platform.claude.com/docs/en/build-with-claude/prompt-caching (response schema)
return {
    "text": text,
    "tool_calls": tool_calls,
    "stop_reason": response.stop_reason,
    "usage": {
        "in_tokens":  response.usage.input_tokens,
        "out_tokens": response.usage.output_tokens,
        "cache_read_tokens":  getattr(response.usage, "cache_read_input_tokens", 0) or 0,
        "cache_write_tokens": getattr(response.usage, "cache_creation_input_tokens", 0) or 0,
    },
}
```
```python
# core/pricing.py — compute_cost signature extension (Claude's Discretion on exact shape)
def compute_cost(model: str, in_tokens: int, out_tokens: int,
                  cache_read_tokens: int = 0, cache_write_tokens: int = 0) -> float:
    pricing = MODEL_PRICING.get(model)
    if pricing is None:
        ...
        return 0.0
    cost = (in_tokens / 1_000_000) * pricing["input"] \
         + (out_tokens / 1_000_000) * pricing["output"] \
         + (cache_read_tokens / 1_000_000) * pricing["input"] * 0.1 \
         + (cache_write_tokens / 1_000_000) * pricing["input"] * 2.0  # 1h TTL multiplier
    return cost
```
The `LLMClient.chat()` cost-metering block (`core/llm_client.py:128-148`) currently only pulls
`in_tokens`/`out_tokens` out of `usage` — it must be extended in the same change to pull the two
new fields and pass them through to both `compute_cost` and `LLMUsageStore.record`.
`LLMUsageStore.record`'s signature (`memory/firestore_db.py:423-443`) and its Firestore schema
comment (lines 410-415) both need the two new `Increment` fields — additive, no migration
required since Firestore is schemaless.

### Pattern 3: TTL-cached store read (BRAIN-07, replicate `SelfStateStore.get()`)

**What:** `SelfStateStore.get()` (`memory/firestore_db.py`, ~lines 500-525) already implements
the exact pattern BRAIN-07 needs — a `_cache_key` tuple set in `__init__`, `_cache_get`/
`_cache_put` around the Firestore read, and `_cache_invalidate_prefix` called from the write
path (`set()`).
**When to use:** `UserProfileStore.load()` currently has NO caching (`memory/firestore_db.py:232-241`)
— confirmed via direct read this session — and is called unconditionally on every
`render_smart_system()` invocation (`core/main.py:313`), i.e. every chat turn and every
autonomous-tick compose. This is the single highest-QPS uncached Firestore read this phase
touches.
**Example:**

### Anti-Patterns to Avoid

- **Passing `temperature`/`top_p`/`top_k` on the Anthropic path when the model is
  `claude-sonnet-5`:** hard HTTP 400, not a soft override. Confirmed via codebase grep this
  session: the ONLY call site anywhere in `core/` that passes an explicit `temperature=` kwarg
  is `core/tick_brain.py:173`, and that targets the Groq/OpenAI backend, not Anthropic — today's
  code is safe as-is, but this must become a documented invariant so a future Anthropic-backend
  call site doesn't add one without checking the model first.
- **Manual extended thinking (`thinking: {"type": "enabled", "budget_tokens": N}`):** removed on
  `claude-sonnet-5`, hard 400 at launch (not a deprecation warning). Do not carry this over from
  any prior Opus-era config.
- **Estimating token counts from character count or `tiktoken`:** `tiktoken` is OpenAI's
  tokenizer family and does not model Anthropic's Sonnet-5 tokenizer at all (~30% more tokens
  for the same text than the prior tokenizer). Use `client.messages.count_tokens(...)` against
  the real `claude-sonnet-5` model string for the BRAIN-06 slimming acceptance criteria.
- **Treating `cache_read_input_tokens` as full-price input:** overstates Klaus's cost dashboard
  by up to 10x on cache-hit calls (reads are 0.1x base price) — directly undermines BRAIN-02's
  "truthful metering" goal.
- **Building a client-side caching wrapper (Redis, in-process LRU) "to implement prompt
  caching":** prompt caching here is a server-side Anthropic feature activated via a request
  field — a client cache would duplicate/conflict with it for no benefit.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|--------------|-----|
| Prompt-prefix caching | A custom in-process response cache keyed on prompt hash | `cache_control: {"type": "ephemeral", "ttl": "1h"}` on the Anthropic request | Server-side Anthropic feature; a client cache duplicates/conflicts and doesn't save Anthropic-side compute or billing the way the native feature does |
| Sonnet-5 token counting for the slimming target | Char-count estimation or a third-party tokenizer (`tiktoken`) | `client.messages.count_tokens(...)` (built into the `anthropic` SDK) | `tiktoken` models OpenAI's tokenizer family, not Anthropic's — numbers would be flatly wrong for a model with a ~30%-different token/text ratio |
| Read-through Firestore caching for `UserProfileStore` | A new caching library or ad-hoc dict on the class instance | The existing module-level `_READ_CACHE`/`_cache_get`/`_cache_put`/`_cache_invalidate_prefix` helpers (`memory/firestore_db.py:61-86`) already used by `SelfStateStore` | Single-user QPS doesn't warrant a new caching abstraction; the existing pattern is already proven and instance-independent (survives across `AgentOrchestrator` singleton lifetime) |
| Daily Groq/Anthropic-spend aggregation | A new BigQuery/analytics pipeline | `LLMUsageStore.summary("today")`/extend with a "yesterday" period, reusing the existing `llm_usage/{YYYY-MM-DD}` doc-per-day schema | The store already accumulates atomic per-day counters via `firestore.Increment` — a tripwire just needs to *read* yesterday's doc, not build new aggregation infra |

**Key insight:** Every piece of infrastructure this phase needs (caching, metering, tokenizer
counting, TTL-cached reads) already exists either natively in the Anthropic SDK or as an
established pattern elsewhere in this exact codebase. The work is disciplined wiring, not new
build — consistent with the milestone-wide research's overall verdict ("the risk … is almost
entirely in wiring discipline and sequencing, not in unknowns").

## Common Pitfalls

*(Full pitfall catalogue with warning signs, recovery steps, and cross-phase mapping lives in
`.planning/research/PITFALLS.md` — these five are the ones that land specifically in Phase 30.5.
Re-verified against live code this session.)*

### Pitfall 1: Tick-brain fallback silently inherits the new (expensive) brain model
**What goes wrong:** `core/tick_brain.py` lines 117-129 build the fallback client from
`os.getenv("SMART_AGENT_BACKEND")`/`SMART_AGENT_MODEL`/`SMART_AGENT_API_KEY` — confirmed still
true as of this research session. The instant those env vars flip to `claude-sonnet-5`, every
Groq hiccup (not just a hard decommission — everyday transient 429s/500s) silently bills Sonnet
pricing, ~10-30x more expensive per token than Gemini Flash was.
**Why it happens:** Model-swap PRs change the primary pointer and treat the fallback as "still
fine" because the fallback *logic* wasn't touched, only the variable it reads.
**How to avoid:** Introduce `TICK_BRAIN_FALLBACK_BACKEND/MODEL/API_KEY/BASE_URL` defaulting to
`gemini`/`gemini-3.5-flash`, ship in the SAME deploy that flips `SMART_AGENT_*`. Never as a
follow-up (D-13's locked ordering).
**Warning signs:** `tick_fallback` purpose entries in LLMUsage show `claude-sonnet-5` instead of
`gemini-3.5-flash`; daily cost jumps with no unusual chat volume, correlated with Groq errors.

### Pitfall 2: Prompt-caching billing fields aren't captured — the new cost tripwire alerts on wrong numbers
**What goes wrong:** `core/llm_client.py:244-247` currently reads only
`response.usage.input_tokens`/`.output_tokens` — confirmed this session. Once `cache_control`
ships, `cache_read_input_tokens`/`cache_creation_input_tokens` are separate response fields; if
not extracted, `compute_cost` either double-counts (treats cache reads as full-price input) or
under-counts (drops them). BRAIN-04's tripwire — the safety net for exactly this class of bug —
alerts on numbers that don't match the actual invoice.
**How to avoid:** Ship the extraction (Pattern 2 above) in the SAME change as `cache_control`
itself, never split across two deploys.
**Warning signs:** LLMUsage cost sum diverges from the Anthropic console by >~10%;
`cache_read_input_tokens` stays 0 on turn 2 of a conversation where the system prompt should be
cache-stable.

### Pitfall 3: Sonnet's literal instruction-following turns Gemini-era prescriptive rules into over-application bugs
**What goes wrong:** `prompts/smart_agent.md` (360 lines, confirmed this session) leans on
`ALWAYS`/`NEVER` imperatives empirically tuned against `gemini-3.5-flash`'s weaker instruction
following. A frontier model executes the same blanket imperatives more literally and more
aggressively — directly opposed to the milestone's "judgment replacing scripts" value.
**How to avoid:** Land the light de-prescription pass (D-06) BEFORE Sonnet takes live traffic;
treat every `ALWAYS`/`NEVER` survivor as a genuine hard invariant (delivery idempotency, cost
gating, secret denylist), move behavioral preferences toward directives/values framing in later
phases. Do NOT touch the Tier A/B contract or persona voice (D-05, verbatim lines ~167-201,
confirmed present at those line numbers this session).
**Warning signs:** Live canary turns feel "stiffer" or more rule-quoting than the Gemini
baseline; Amit reports Klaus reciting a rule instead of acting/observing.

### Pitfall 4: `max_tokens` truncation once adaptive thinking is on by default
**What goes wrong:** `core/llm_client.py:36` currently sets module-level `MAX_TOKENS = 4096`.
Adaptive thinking (default `effort: "high"` when `thinking` is omitted on Sonnet 5) consumes the
same `max_tokens` budget as the visible response — a high-effort turn can spend most of a 4096
budget on thinking and return `stop_reason: "max_tokens"` with a truncated or empty visible
answer.
**How to avoid:** Raise the Anthropic-path default to ~16K per the locked D-11 decision, leaving
real headroom beyond expected response length. Leave the tick-brain's 2048 (Groq admission
control, unrelated concern) and any deliberately small ceilings untouched.
**Warning signs:** `stop_reason: "max_tokens"` on turns with no visible text; empty/truncated
Sonnet replies immediately post-cutover.

### Pitfall 5: `deploy.yml --set-env-vars` clobbers out-of-band Cloud Run env vars
**What goes wrong:** The live deploy workflow (`.github/workflows/deploy.yml` line 100, confirmed
this session) passes a single giant `--set-env-vars` string with every current env var
enumerated. Any new var (`TICK_BRAIN_FALLBACK_*`, `KLAUS_DAILY_COST_ALERT`,
`SMART_AGENT_TERTIARY_*` if that shape is chosen) added only via the Cloud Run console — not
also added to this workflow string — is silently wiped on the next routine deploy (documented
project memory: `klaus_deploy_retrigger`/`klaus_web_push_gotchas`).
**How to avoid:** Every new env var this phase introduces MUST be added to the `deploy.yml`
`--set-env-vars` string (or `--update-secrets` for API keys), never set only via console.
**Warning signs:** A feature that worked in manual testing regresses after the next unrelated
deploy with no code change.

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|-------------------------|
| Fallback API key confusion (wrong secret backing a fallback tier post-rename) | Tampering / Information Disclosure (wrong model receiving traffic, potential cost/data exposure) | Verify each `--update-secrets` mapping in `deploy.yml` explicitly post-deploy by checking the live Cloud Run revision config, not just the workflow YAML (per the documented deploy-clobber gotcha) |
| Malformed `KLAUS_DAILY_COST_ALERT` crashing the heartbeat cron | Denial of Service (of the monitoring function itself) | Defensive float-parse with a hardcoded default, matching the existing `TICK_BRAIN_MAX_TOKENS`/`TICK_BRAIN_TEMPERATURE` try/except pattern in `core/tick_brain.py:95-106` |
| Cost-tripwire compose call itself becoming an unbounded-cost loop (a Sonnet call composing an alert about Sonnet costs) | Denial of Service (cost) | Once-per-day suppression (D-01) already bounds this to at most one paid compose call per day; deterministic template fallback (D-04) if the compose call fails avoids retry storms |
