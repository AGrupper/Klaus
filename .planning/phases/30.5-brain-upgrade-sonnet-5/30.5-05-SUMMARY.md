# Phase 30.5 Plan 05: Heartbeat Daily-Spend Tripwire Summary

**A once-daily, per-date-suppressed heartbeat tripwire that reads yesterday's LLM spend, brain-composes a Klaus-voiced narrative alert (Sonnet, `purpose="cost_tripwire"`) naming the top cost drivers and cache-hit rate, with a deterministic plain-text fallback and send-gated suppression marking.**

## Performance

- **Duration:** ~20 min
- **Completed:** 2026-07-17
- **Tasks:** 2 (Task 1 TDD: RED test commit -> GREEN implementation commit; Task 2 plain auto)
- **Files modified:** 3 (1 created, 2 modified)

## Accomplishments
- `core/heartbeat.py::check_daily_spend()` reads `LLMUsageStore.summary_for_date(yesterday)`, defensively parses `KLAUS_DAILY_COST_ALERT` (mirroring `tick_brain.py`'s try/except float-parse, default $5.0), and gates on `CostTripwireLogStore.already_fired(yesterday)` — returns `None` unless yesterday's `total_cost_usd` exceeds the threshold and the tripwire hasn't already fired for that date.
- `_compose_spend_alert()` brain-composes a Klaus-voiced alert via `SMART_AGENT_*` (Sonnet) reading the new `prompts/cost_tripwire.md`, tagged `purpose="cost_tripwire"` so the alert's own cost is auditable (T-30.5-15), with `_spend_plain_text_fallback()` as a deterministic template fallback on any compose failure (D-04).
- `_send_daily_spend_alert()` wires `check_daily_spend()` into `run_tick()` as a separate guarded step (not inside `_collect_signals` — it's a once-daily paid-compose-and-send action, not a `Signal`-emitting ops checker), sends via the existing `send_and_inject` path, and calls `CostTripwireLogStore.mark_fired()` only after a successful send (mirrors the `OutreachLogStore` D-10 gating — a delivery failure leaves `already_fired` False so it retries on the next hourly tick).
- `prompts/cost_tripwire.md` instructs the model that all numbers (total cost, top cost drivers, cache-hit rate, threshold) are pre-computed and supplied — it must phrase them, never recompute or invent, and give Klaus's own read on normal-vs-broken usage (D-03).

## Decisions Made
- `check_daily_spend()` deliberately does not send or mark_fired itself — it returns `{"date", "text", "summary"}` or `None`, keeping the pure compose logic separately testable from the send-then-suppress side effect, which `_send_daily_spend_alert()` (Task 2) owns.
- The tripwire check is unconditional on every hourly tick rather than restricted to a specific hour window, per the plan's explicit note that per-date suppression makes the exact trigger hour irrelevant.
- No quiet-hours deferral was added for the spend-tripwire send — the plan's `must_haves` cover per-date suppression only; adding quiet-hours gating would have been unrequested scope beyond D-01..D-04.
- `prompts/cost_tripwire.md`'s voice follows the currently-live `docs/AGENT.md` persona (plain prose, no "Sir", dry directness) rather than the plan's literal "JARVIS register" wording, since `docs/AGENT.md`'s actual content has evolved past that label (CLAUDE.md's file-reference table retains the old description text).
