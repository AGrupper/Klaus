## Critical Issues

### CR-01: Mid-stream transport errors escape LLMError mapping — 3-tier fallback chain bypassed

**File:** `core/llm_client.py:248-278`
**Issue:** The always-stream change replaced `messages.create` with `messages.stream(...)` but kept only two exception conversions: `anthropic.APIStatusError` and `anthropic.APIConnectionError` → `LLMError`. In the anthropic SDK (0.105.2, verified in `.venv`), transport exceptions raised **while iterating the SSE body** are not wrapped: `Stream._iter_events` yields straight from `response.iter_bytes()` with no try/except (`anthropic/_streaming.py:74-76`), so a stalled or dropped connection mid-generation raises raw `httpx.ReadTimeout` / `httpx.ReadError` / `httpx.RemoteProtocolError` out of `stream.text_stream` / `get_final_message()`. Under the old non-streaming path these surfaced at request level and were SDK-wrapped as `APITimeoutError` (a subclass of `APIConnectionError`) → `LLMError`. Consequences:
- `_run_smart_loop` (core/main.py:638) catches only `LLMError` — a mid-stream timeout **bypasses the entire Sonnet→Gemini→Haiku chain and `CONNECTIVITY_ERROR_TEXT`** and escapes as an unhandled exception; the turn hard-fails with no fallback.
- The code comment ("hung connections still die" via the read timeout) explicitly relies on exactly this mid-stream `httpx.ReadTimeout` — i.e., the intended timeout mechanism is guaranteed to take the unmapped path.
- Compose call sites (`morning_briefing`, `nightly_review`, `weekly_training_review`, `heartbeat._compose_spend_alert`, `reflection`) are shielded by broad `except Exception`, so only the conversational turn path is exposed — but that is the highest-traffic path.

**Fix:**
```python
import httpx  # transitive dep of anthropic; top-of-method with the anthropic import

try:
    ...  # both stream branches unchanged
except anthropic.APIStatusError as exc:
    raise LLMError(str(exc), backend="anthropic", status_code=exc.status_code) from exc
except anthropic.APIConnectionError as exc:
    raise LLMError(f"Connection error to Anthropic API: {exc}", backend="anthropic") from exc
except httpx.HTTPError as exc:
    # Mid-stream transport failure (ReadTimeout/ReadError/RemoteProtocolError):
    # the SDK does not wrap exceptions raised while iterating the SSE body.
    raise LLMError(f"Anthropic stream transport error: {exc}", backend="anthropic") from exc
```
Callback exceptions (`TurnCancelled` from `on_text_delta`) are not `httpx.HTTPError`, so the D-19 "propagate untouched" contract is preserved.

## Warnings

### WR-01: Daily-spend tripwire bypasses quiet hours — deterministically fires around midnight

**File:** `core/heartbeat.py:1001-1026, 1037`
**Issue:** `_send_daily_spend_alert` calls `send_and_inject` with no `_in_quiet_hours` gate, unlike every other heartbeat send (critical signals are queued via `_queue_signals` during quiet hours, core/heartbeat.py:1060-1064). Because `check_daily_spend` evaluates **yesterday's** usage doc and is suppressed per-date, the first eligible tick after a >$5 day is the 00:00 hourly tick — squarely inside the 22:00–07:00 quiet window. The docstring's claim that "the exact trigger hour [is] irrelevant" is inverted: per-date suppression makes the trigger hour *deterministic*, and it is always a nighttime push. `send_and_inject` (core/scheduled_message.py) has no quiet-hour handling of its own.
**Fix:** Gate the send on `not _in_quiet_hours(config, now)` (pass `config`/`now` into `_send_daily_spend_alert`); the per-date suppression already guarantees it will fire at the first post-07:00 tick instead, with no re-fire risk.

### WR-02: Blocking Sonnet compose runs on the event loop inside the heartbeat cron

**File:** `core/heartbeat.py:807-855` (called from `core/heartbeat.py:1015` inside `async _send_daily_spend_alert`)
**Issue:** `check_daily_spend()` → `_compose_spend_alert()` performs synchronous Firestore reads plus a blocking Sonnet `client.chat(...)` (streaming, up to `LLM_TIMEOUT_SECONDS=120s`) directly inside an `async` function awaited by `run_tick`, which `/cron/heartbeat` awaits on the event loop (`interfaces/web_server.py:911`). This is the exact Pitfall-2 class behind the weekly-review 500 incident (fixed then via `run_in_executor`) — while the compose runs, every concurrent request on the instance (hub turns via `/internal/process-update`, webhook ACKs) starves. Other cron routes consistently use `loop.run_in_executor(None, ...)` for blocking batch work. The pre-existing `_run_tick_brain_pass` shares this flaw, but that call is a fast free-tier Groq hop; this phase adds a potentially 120s paid Sonnet generation to the same loop.
**Fix:** `alert = await asyncio.get_running_loop().run_in_executor(None, check_daily_spend)` in `_send_daily_spend_alert`.

### WR-03: 3-tier chain re-attempts a dead primary on every tool iteration — no sticky fallback or turn deadline

**File:** `core/main.py:630-685` (HEAD)
**Issue:** The fallback chain is evaluated per iteration of the tool loop (`MAX_TOOL_ITERATIONS = 12`). When the primary is down in a *slow* mode (hangs/timeouts rather than fast 401s), each iteration pays up to ~240s of primary latency (client `max_retries=1` → 2 attempts × 120s read timeout) before the Gemini tier even runs; a tool-heavy turn can therefore stretch into tens of minutes while "working". If the fallback also times out, the tertiary adds up to another ~240s per iteration before `CONNECTIVITY_ERROR_TEXT`. The 2-tier version of this pattern predates the phase, but Plan 06 both deepened the chain and flipped the primary to the backend whose observed latency already sits at 70s/120s headroom (D-14 item 3).
**Fix:** After the first primary `LLMError` in a turn, set a local `active_client = self.smart_agent_fallback` (sticky for the remainder of that turn, with the D-12 disclosure appended once on the final text), or at minimum record `primary_failed = True` and skip straight to the fallback on subsequent iterations.

### WR-04: Dated Sonnet-5 snapshot IDs silently meter $0

**File:** `core/pricing.py:51, 64-68`
**Issue:** `_pricing_for` special-cases only the exact string `"claude-sonnet-5"`. `claude-haiku-4-5` has a dated alias row (`claude-haiku-4-5-20251001`) precisely because Anthropic model pins drift to snapshot IDs, but no equivalent exists for Sonnet-5. If `SMART_AGENT_MODEL` is ever pinned to a snapshot (e.g. `claude-sonnet-5-20260601` — a plausible ops move given the model map is env-driven by design), `compute_cost` logs once and returns 0.0 forever: BRAIN-02 cost transparency and the BRAIN-04 $5 tripwire both go silently blind on the most expensive model in the system.
**Fix:** Prefix-match in `_pricing_for`:
```python
if model == _SONNET_5_MODEL_ID or model.startswith(_SONNET_5_MODEL_ID + "-"):
    return _sonnet_5_pricing()
```

## Info

### IN-01: Tertiary tier shares the primary's API key — diversity hedge weakened

**File:** `.github/workflows/deploy.yml:101` (`SMART_AGENT_TERTIARY_API_KEY=klaus-anthropic-key:latest`)
**Issue:** Tiers 1 and 3 ride the same Anthropic key/account. Auth failures, billing lapses, and account-level rate limits — the most common real-world Anthropic failures, including the exact bogus-key scenario D-14 tested — take out both Anthropic tiers together; the tertiary only covers Sonnet-model-specific outages. Acceptable with one Anthropic account, but worth recording that the "3-tier" chain is effectively 2 failure domains.
**Fix:** None required now; note it in Phase 35 hardening (separate key, or accept and document).

### IN-02: SELF.md Model Map omits the tertiary (Haiku) tier

**File:** `core/self_manifest.py:483-490`
**Issue:** The Model Map renders brain / worker / fallback / tick-brain / tick-brain-fallback / embeddings, but not the newly configured `SMART_AGENT_TERTIARY_*` tier. Klaus's self-manifest no longer fully describes his own live fallback chain — drift-by-omission against the "SELF.md can never drift" invariant this generator exists to enforce.
**Fix:** Add a `| Smart agent tertiary | {os.getenv("SMART_AGENT_TERTIARY_MODEL", "claude-haiku-4-5")} | ... |` row reading the env like the other rows.

### IN-03: UserProfileStore caches negative results and bootstrap never invalidates

**File:** `memory/firestore_db.py:233-252, 272-292` (HEAD)
**Issue:** `load()` caches `{}` when the doc is absent, and `bootstrap_if_empty()` writes the scaffold without calling `_cache_invalidate_prefix`. A `load()` that races ahead of bootstrap on a cold instance serves stale `{}` for up to 10 minutes after seeding. One-time cold-start edge (the doc exists permanently once seeded), but free to fix.
**Fix:** Invalidate the cache after the scaffold `set()` in `bootstrap_if_empty`, mirroring `update()`.

### IN-04: Cached profile returned via shallow copy — nested state is shared

**File:** `memory/firestore_db.py:243, 251` (HEAD)
**Issue:** `dict(cached)` copies only the top level; nested dicts/lists in the profile scaffold are shared references across all callers and the cache itself. Any caller mutating nested state poisons the cache until TTL/invalidation. Consistent with the pre-existing pattern used by the other cached stores, but the profile is the store most likely to hold nested structures.
**Fix:** `copy.deepcopy(cached)` (profile is small; cost is negligible at this read rate), or document the read-only contract.

### IN-05: Morning-briefing composes meter as "unknown" cost driver

**File:** `core/morning_briefing.py:527-548`
**Issue:** Both the primary and new fallback compose calls omit `purpose=`, so `LLMUsageStore.record` buckets them into `unknown_calls`/`unknown_cost_usd`. The BRAIN-04 tripwire's "top 2-3 cost drivers by purpose" will attribute morning-briefing Sonnet spend to "unknown" — a self-inflicted blind spot in the very feature this phase built. (Deliberate per the Plan 06 summary decision, recorded here as a follow-up seed.)
**Fix:** Pass `purpose="morning_briefing"` on both calls.

### IN-06: `MODEL_PRICING["claude-sonnet-5"]` frozen at import time and aliases the intro dict

**File:** `core/pricing.py:51`
**Issue:** The row is evaluated once at import (`_sonnet_5_pricing()`), so a long-lived process crossing the 2026-08-31 cutoff carries a stale display row, and the row is the *same object* as `_SONNET_5_INTRO_PRICE` — any mutation of one mutates the other. Compute paths are unaffected (`_pricing_for` resolves live and is the only internal reader), and the inline comment flags it, but the row is a trap for any future direct `MODEL_PRICING` reader.
**Fix:** Store a sentinel/copy (`dict(_sonnet_5_pricing())`) or drop the row and let `_pricing_for` be the single source.

### IN-07: Failed tripwire send re-pays the Sonnet compose every hourly tick

**File:** `core/heartbeat.py:807-855, 1001-1026`
**Issue:** `mark_fired` is correctly gated on send success (D-10 pattern), but the paid `_compose_spend_alert` runs *before* the send inside `check_daily_spend` — during a delivery outage the same alert is re-composed (and re-billed) up to ~15×/day until a send lands. Bounded and cheap relative to the >$5 day that triggered it, but it is a cost-tripwire that spends money retrying.
**Fix:** Cache the composed text keyed by date (Firestore or module-level) or compose lazily only on first attempt per date.

### IN-08: D-12 disclosure edge cases on tool-call and empty fallback responses

**File:** `core/main.py:660` (HEAD)
**Issue:** `response["text"] = (response.get("text") or "") + FALLBACK_DISCLOSURE_TEXT` runs unconditionally on fallback success: (a) when the fallback response carries `tool_calls`, the disclosure lands in the discarded tool-loop preamble and may be appended again on later fallback iterations (the model then sees prior disclosures in-history and may echo them); (b) when Gemini returns empty text with no tool calls, the user's entire visible reply is just the italic disclosure line. The ~79-char disclosure stays under the 100-char exhaustion guard (core/main.py:786), so it cannot alone trigger the double-send suppression — no deeper interaction found.
**Fix:** Append only when `not response["tool_calls"]` (final-text branch), or guard with a per-turn `disclosure_added` flag.

### IN-09: Hardcoded "9 scheduled jobs" count in the compact cron summary

**File:** `core/self_manifest.py:520`
**Issue:** The compact cron section hardcodes both the job list and its count as one literal string. The old table was equally hand-maintained, but the count adds a second thing to forget — the next cron add/retire will make SELF.md claim the wrong number even if the list is updated.
**Fix:** Build the summary from a list literal and interpolate `len(jobs)`.
