## Spacing Scale

Reinterpreted for Telegram message structure. "Spacing" = blank lines and section separators.

| Token | Telegram Equivalent | Usage |
|-------|---------------------|-------|
| xs | No separator (inline) | Continuation on same line: `RPE: 7 · notes: "tough" ` |
| sm | Single newline | Between list items within a section |
| md | Blank line (double newline) | Between scorecard rows in the weekly review |
| lg | `---` horizontal rule | Between major sections (same pattern as `morning_briefing.md`) |
| xl | Section header emoji + bold label | Marks named sections (📅 Schedule, ✅ Tasks) |

**Physical spacing exceptions:** none. Follow the existing `---` rule separator convention from `prompts/morning_briefing.md`.

**Source:** Existing `prompts/morning_briefing.md` structure (pre-populated).

## Typography

Reinterpreted for Telegram: "typography" = markdown formatting and emphasis conventions.

| Role | Telegram Equivalent | Weight Analog | Usage |
|------|---------------------|---------------|-------|
| Body | Plain text | Regular | Narrative paragraphs in briefings, recovery framing, weekly review prose |
| Label | `*bold*` (Telegram MarkdownV2) | Semibold | Section headers, metric names in the scorecard (`*HRV:*`, `*ACWR:*`) |
| Emphasis | `_italic_` (MarkdownV2) | Italic | JARVIS dry-wit asides; caveat notes (e.g. thresholds are v0 heuristics) |
| Code/data | `` `monospace` `` | Monospace | Raw numbers only when precision matters (e.g. `ACWR 1.62`); never for layout |

**Rules:**
- Never use monospace tables — alignment breaks on mobile Telegram (D-18, scoping note).
- Bold is for labels only; do not bold entire sentences.
- No emojis in prose body (AGENT.md directive). Emojis appear only in structural positions: section headers and scorecard status indicators.
- No exclamation marks anywhere (AGENT.md directive).
- Address Amit as "sir" in all automated messages.

**Source:** `docs/AGENT.md` (tone directives), `prompts/morning_briefing.md` (existing conventions), D-18 (no monospace tables).

## Color

Reinterpreted for Telegram: "color" = emoji semantic palette. These emoji carry the visual weight that colors carry in a web UI.

| Role | Emoji | Context | Semantic Meaning | Usage |
|------|-------|---------|-----------------|-------|
| Dominant (neutral) | — | all | Plain text background | Body copy, no emoji decoration |
| Positive / done | ✅ | scorecard row | Completed | Weekly scorecard: workout logged (any source) |
| Negative / missed | ❌ | scorecard row | Skipped | Weekly scorecard: workout explicitly skipped |
| Warning / unknown | ⚠️ | scorecard row | No log entry | Weekly scorecard: planned workout with no record — not confirmed done or skipped |
| Recovery concern (mild) | ⚠️ | recovery preamble | Mild caution | Recovery framing preamble when severity = mild |
| Recovery concern (strong) | 🔴 | recovery preamble | Strong caution | Recovery framing preamble when severity = strong (D-12) |
| Section navigation | 📅 📧 ✅ 📚 🥗 | section header | Header anchors | Pre-rendered section markers; same set as `morning_briefing.md`; do not invent new section emoji |

**Context disambiguation:** ⚠️ means "no log entry" only in a scorecard row, and "mild recovery concern" only in a recovery preamble — the two never share a message context. ✅ likewise means "workout completed" in a scorecard row and is a structural header anchor in the section-navigation set; never mix the two roles within one line.

**Accent reserved for:** scorecard status column only (✅/❌/⚠️ per workout row). Do not scatter status emoji into prose.

**Source:** D-18 (✅/❌/⚠️ scorecard), D-12 (severity levels), `prompts/morning_briefing.md` (existing section emoji).

## Copywriting Contract

### Voice
JARVIS × C-3PO: precise, composed, dry — never fawning or corporate. "Sir." No exclamation marks. No emojis in prose. Metric-anchored where data is present. Suggesting (not commanding) in recovery contexts. Empty profile → no invented personal targets (D-13 / PROMPT-02 guardrail).

**Source:** `docs/AGENT.md`, D-13, PROMPT-02.

### Inline Keyboard Layouts

#### RPE Picker (CHECKIN-04, D-26)
Two rows of five buttons. Scale anchor in the prompt text immediately above the keyboard.

```
Prompt text: "How did [workout name] feel? Rate your effort: 1 = easy · 10 = max effort"

Row 1:  [ 1 ]  [ 2 ]  [ 3 ]  [ 4 ]  [ 5 ]
Row 2:  [ 6 ]  [ 7 ]  [ 8 ]  [ 9 ]  [ 10 ]
```

- Button callback data format: `rpe:{session_key}:{value}` (implementation detail for executor).
- After tap: RPE is logged immediately; follow-up notes prompt sent in a new message.
- Scale anchor ("1 = easy · 10 = max effort") must appear in the prompt text, not inside a button.

**Source:** D-26 (two rows of 5), CHECKIN-04.

---

#### Watch-Off Branch (D-08)
Shown when Garmin has no activity record for a planned workout.

```
Prompt text: "No Garmin record for [workout name]. What happened?"

[ Did it — watch was off ]   [ Skipped ]
```

- "Did it — watch was off" → triggers RPE picker (D-08).
- "Skipped" → triggers skip-reason keyboard.
- Callback data: `watchoff:{session_key}:done` / `watchoff:{session_key}:skipped`.

**Source:** D-08.

---

#### Skip-Reason Buttons (D-08b)
Shown after user taps "Skipped" in the watch-off branch.

```
Prompt text: "Got it. What was the reason?"

[ Rest / recovery ]   [ Sick / injured ]
[ Too busy ]          [ Other — tell me ]
```

- First three buttons: one-tap structured log. Callback: `skipreason:{session_key}:{reason_key}`.
  - Reason keys: `rest_recovery`, `sick_injured`, `too_busy`.
- "Other — tell me": triggers free-text note request. Same open-note flow as the notes step.
- Structured reason enables weekly review surfacing (D-08b intent).

**Source:** D-08b.

---

#### Notes Follow-Up (D-05)
Sent after RPE is logged. No inline keyboard — open text input.

```
"Logged — RPE {N}. Anything to note about that session? Reply to this message, or /skip."
```

- Note attaches when the user replies-to this specific message (native Telegram reply).
- If user sends free text without replying-to: brain decides from injected pending-note context whether it is the note (D-05).
- `/skip` or no reply: RPE-only entry is complete and valid.
- No auto-close timer (D-05: "open until you reply").

**Source:** D-05.

### Message Copy — Weekly Training Review (REVIEW-03, D-17..D-24)

**Structure:**

```
Good morning, sir. Here is your training review for the week ending [Sat date].

[Scorecard — one line per planned workout]
✅ Monday — [Type] — RPE [N]
❌ Wednesday — [Type] — [skip reason if known]
⚠️ Friday — [Type] — no log entry

[2–4 paragraphs of reflective coaching narrative]
• HRV trend: [this week avg] → [last week avg]  ↑/↓
• RHR trend: [this week avg] → [last week avg]  ↑/↓
• Sleep: [this week avg] → [last week avg]  ↑/↓

[One suggestion grounded in this week's actual data, JARVIS voice, no invented targets]
```

- Scorecard format: no monospace table; plain emoji + text rows (D-18).
- Narrative wraps around the scorecard (D-19): coaching analysis, trend lines, one suggestion.
- One suggestion: data-grounded, direct, no fabricated HR zones/weights/paces (D-20).
- Nutrition: raw `MealStore` 7-day totals in the narrative, not a separate section (D-21).
- Trend window: this week vs. last week, expressed as arrows or directional words (D-22).
- Week boundary: previous Sun–Sat (D-23).
- Empty-log weeks: always send; "Quiet week — not enough data to review yet, sir." note (D-24).

**Source:** D-17 through D-24.

### Message Copy — Recovery Concern (D-12, D-13, D-16)

Recovery concern appears in BOTH `morning_briefing.md` summary line AND `proactive_alert.md` body.

**Mild severity:**
- Tone: direct, suggesting, metric-anchored. One sentence woven into the existing summary/alert.
- Example: "ACWR is at 1.6 and sleep was below par last night — might be worth keeping today's session submaximal, sir."
- Never: commands ("you must dial back"), invented numbers (specific weights, HR zones, paces).

**Strong severity:**
- Tone: same JARVIS voice, more direct. Explicitly names the combination of signals.
- Example: "ACWR is at 1.8, HRV is flagged unbalanced, and that's two rough nights in a row — genuinely recommend dropping a set or two and avoiding high-intensity work today, sir."
- 🔴 prefix on the recovery note in the proactive alert (not in the morning briefing prose).

**Guardrail (D-13):** With an empty `UserProfileStore`, Klaus names qualitative modifications only — never invents a specific weight, HR cap, or pace target. Profile-empty path: "drop a set or two" / "keep it submaximal" / "favour aerobic over anaerobic today."

**No recovery concern:** Usual briefing/alert — no recovery framing at all. Do not add "all clear" or "recovery looks good" placeholder.

**Source:** D-12, D-13, D-16, RECOVERY-03.

### Message Copy — Empty / Error States

| State | Copy |
|-------|------|
| Training calendar not found | Silent (check-in skips cleanly; heartbeat/logs the error internally) |
| Garmin data unavailable during check-in | Treat all planned workouts as unverified → prompt watch-off branch for each |
| Garmin data unavailable during weekly review | "Garmin data was unavailable for this review, sir. Training log entries are shown; biometric trends could not be computed." |
| TrainingLogStore read failure | "I was unable to retrieve the training log for this review, sir. The weekly review will retry next Sunday." |
| Weekly review — quiet week / no data | "Quiet week — the Training calendar shows no sessions, and the log has no entries. Nothing to review, sir." (D-24: still sent) |
| Notes step — pending note expires (future scope) | N/A in Phase 20 (no carry-over per deferred section) |
| RPE button tap — session not found | "I couldn't match that button to an open session, sir. The log entry may have already been closed. Reply with your RPE and I'll update the log." |

---

### Message Copy — Destructive Actions

No destructive actions in this phase. `TrainingLogStore` entries are additive; log edits are handled via `log_training` brain-direct tool in conversation (not via inline keyboard). No delete/clear flows.

## Interaction / State Flow Summary

```
21:30 proactive-alerts cron fires
  ├─ Garmin silent sync → TrainingLogStore (no Telegram)
  ├─ For each unlogged planned workout (start time < now):
  │   ├─ Garmin record exists, RPE present → silent log (source=garmin)
  │   ├─ Garmin record exists, RPE absent → RPE picker keyboard
  │   │   └─ tap → log RPE immediately → notes prompt (open until reply or /skip)
  │   └─ No Garmin record → watch-off branch
  │       ├─ "Did it — watch was off" → RPE picker → notes prompt
  │       └─ "Skipped" → skip-reason buttons (4 options)
  │           └─ "Other" → free-text notes prompt
  └─ All workouts covered → silent (CHECKIN-05)

Sunday 10:00 weekly-review cron fires
  └─ Compose → send scorecard + narrative → always (D-24)

Morning briefing tick (existing, extended)
  └─ recovery_concern computed → tone shift injected into summary line if set

21:30 proactive-alerts (existing, extended)
  └─ recovery_concern flag → recovery framing added to alert body if set
```
