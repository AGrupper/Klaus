# Phase 30 — UI Design Contract: Health Pages

> Visual and interaction contract for Phase 30. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> This phase **extends** the existing Klaus Hub design system built in Phases 26–29 — every reused token,
> spacing value, and component pattern below is sourced from the live codebase
> (`frontend/src/tokens.ts`, Phase 26/27/28 components). New additions (chart colors, chart primitives,
> log-row exceptions) are called out explicitly and are the first genuinely new frontend infrastructure
> added since the shell (no chart library exists today — see Design System § Chart Library).

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Tailwind v4 utility-first; no shadcn) — same as Phases 26–28 |
| Preset | not applicable |
| Component library | none (hand-rolled, inline styled via `frontend/src/tokens.ts` constants) |
| Icon library | lucide-react 1.18.0 (already a dependency) |
| Font | `system-ui, -apple-system, "Segoe UI", sans-serif` — no web font download |
| Chart library | **none — custom lightweight inline SVG chart primitives** (new: `frontend/src/components/charts/`) |

No `components.json` found; shadcn initialization gate does not apply (extension of an existing hand-rolled
system, not a greenfield build — same as Phase 28/27).

**Chart library decision (D-04 discretion):** Klaus Hub's established pattern is hand-rolled visuals with
zero new dependencies where feasible — `ContributionGrid.tsx` (Phase 28) explicitly avoids a heatmap library
("Pure CSS `display:grid` — no chart/heatmap library"). This phase needs only line charts (with gaps,
a dashed reference line, and a dual-series overlay), one bar-over-line combo (sleep score + duration), and
tap/hover point tooltips — no zoom/pan/scrubbing (D-04 explicitly rejects this). A ~90KB+ dependency
(Recharts) or a d3-wrapper (visx) is disproportionate to this scope and adds PWA precache weight for a
single phase. **Decision: build a small internal SVG chart toolkit** (`LineChart`, `BarChart`,
`ChartTooltip`) shared across all three pages, matching the dark-theme tokens exactly and giving full
control over D-08's gap-rendering rule (no line library handles "never interpolate, never zero-fill"
out of the box without fighting its API). This keeps `frontend/package.json` dependency-free for this
phase and is consistent with the existing hand-rolled philosophy.

## Spacing Scale

All values reused from Phase 26/27/28 (`frontend/src/tokens.ts` conventions + established inline patterns).
No changes to the base scale.

| Token | Value | Usage in Phase 30 |
|-------|-------|--------------------|
| xs | 4px | Icon-to-label gaps in modality badges and chip rows |
| sm | 8px | Intra-row gaps in log entries; chart legend item gaps |
| md | 16px | Chart card padding; page content padding; section gaps |
| lg | 24px | Space between the trend-chart row and the log list (Training page) |
| xl | 32px | Not used in this phase |
| 2xl | 48px | Not used in this phase |
| 3xl | 64px | BottomTabs height (carry-over invariant — do not modify) |

Exceptions:
- **Touch targets:** minimum 44px on all interactive elements on phone (iOS HIG) — applies to sub-tab
  buttons, range-toggle buttons, macro chips, log entry rows (tap-to-drill-down), and grid cells collectively
  (see grid note below).
- **Log entry row height:** `minHeight: 64px` (vs. 52px for `HabitRow`/`TaskRow`) — training log rows carry
  more information (date, modality badge, summary line, chevron) than a habit/task row. Named exception,
  multiple of 4.
- **Modality left-border stripe:** 4px width — reuses the existing "accent left-border stripe" convention
  (`DueTasksBand`/`HabitsBand`) but colored per modality instead of accent (see Color § Training modality
  colors below).
- **Slot-adherence grid cell:** 12px × 12px, 2px gap — reuses the `ContributionGrid` convention exactly
  (Phase 28 named exception; do not round the 2px gap up to 4px, it destroys the density aesthetic).
- **Chart height:** 160px (phone) / 220px (desktop) per chart card — a rendering dimension, not a spacing
  token, declared here for completeness. Chart width is fluid (100% of card content width).
- **Chart card padding:** 16px (md token) all sides, consistent with existing card conventions
  (GlanceRail cards, HabitRow cards).
- **Block divider row padding:** `10px 14px` — mirrors the existing band-header padding pattern
  (`DueTasksBand`/`HabitsBand`, `10px 14px 6px`), simplified to a single-line divider (no icon row beneath).

## Typography

Locked by Phase 26 UI-SPEC. Exactly 4 sizes, exactly 2 weights. Source: `frontend/src/tokens.ts`. No new
sizes or weights introduced in this phase.

| Role | Size | Weight | Line Height | Usage in Phase 30 |
|------|------|--------|-------------|--------------------|
| Display | 28px | 600 (semibold) | 1.15 | Not used in this phase |
| Heading | 20px | 600 (semibold) | 1.2 | Not used at the page-chrome level (no redundant "Health" title — the sub-tab bar is the in-page nav, same principle as TasksPage/HabitsPage not repeating their nav label). Reserved if a page ever needs a true section heading. |

**Weight-only distinction precedent (carried from Phase 28):** where a value needs to stand out at the same
font size as its surrounding metadata, use weight 600 instead of introducing a new size. Applied here to:
active sub-tab/range-toggle text (13px/600 vs 13px/400 inactive), chart tooltip values (16px/600),
and header stat row values (16px/600, mirrors the existing `NutritionStrip` value styling in
`TimelineHeader.tsx`).

No intermediate weights (500 is not used). No font sizes outside the declared set of 13/16/20/28.

## Interaction Contracts

### Page structure & navigation (D-01, D-02, D-03)

- `HealthPage` renders `SubTabs` at the top of the standard center column — **no layout exception**: desktop
  keeps sidebar + Health content + GlanceRail + collapsible DockChat exactly as on Today/Tasks (D-03).
- Switching sub-tabs is a pure client-side state change (no route change, no full remount penalty beyond
  react-query's normal per-key caching) — write the active tab to `localStorage['health-tab']` on every
  change so a reload lands back on the same sub-tab (D-02). First-ever visit (no localStorage key) defaults
  to `'training'`.
- Each sub-page owns its own `RangeToggle` state (`useState<'7d'|'30d'|'90d'|'1y'>('30d')`) — **not**
  persisted across navigations or reloads (D-06: default is always 30 days on page open).

### Range toggle → data fetching

- Changing the range toggle triggers a new `useQuery` with key `['health', 'training' | 'nutrition' |
  'sleep', range]` via `apiFetch`, mirroring the existing `useToday`/`useTaskSummary` react-query pattern.
- `staleTime: 5 * 60 * 1000` (5 min) — historical data doesn't change intraday the way Today's timeline does;
  `refetchOnWindowFocus: true` still applies so a return-to-tab picks up any newly-synced cron data.
- Loading state: `Skeleton` blocks (existing shared component) replace chart cards and log rows during the
  initial fetch for a given range — same visual language as Today timeline's skeleton state (HUB-03).
- Backend returns pre-aggregated points (daily for ≤90d, weekly for >90d per D-07) — the frontend **never**
  re-buckets client-side; `LineChart`/`BarChart` render whatever `{x, y}` points arrive, time-scaled on the
  x-axis so daily and weekly spacing both render correctly without special-casing.

### Chart tap/hover tooltip (D-04)

- **Desktop:** hovering near a point/bar shows `ChartTooltip` at that x-position; moving away hides it.
- **Phone:** tapping near a point/bar shows `ChartTooltip`; it persists until the user taps elsewhere on the
  chart or taps a different point. No pinch-zoom, no pan, no scrubbing gesture (D-04 explicit rejection).
- Nearest-point hit-testing: find the closest x-value within the series to the pointer/touch x-coordinate;
  no minimum distance threshold beyond the chart's own point spacing.
- Tapping/hovering an x-position that falls on a data gap (D-08) shows `ChartTooltip` with the date and
  "No data" in textSecondary — the tooltip still appears (so gaps are discoverable), it just carries no value.

### Missing-data gaps (D-08, cross-cutting)

- A `null` value at any point in a series renders as a visible break in the line/bar — never a zero, never
  an interpolated bridge across the gap. This mirrors `fetch_nutrition_trend`'s `missing_dates` semantics
  exactly (unlogged day ≠ zero-calorie day; watch-not-worn ≠ HRV of 0).
- `LineChart` implementation note: split the SVG path into multiple `<path>` segments at every `null`,
  rather than a single path with `moveTo` gaps that some renderers still visually connect.

### Training log drill-down (D-10, D-12)

- Tapping any `TrainingLogEntry` opens its matching drill-down sheet (`StrengthDrilldownSheet` /
  `RunDrilldownSheet` / `BenchmarkDrilldownSheet`) — same bottom-sheet-on-phone/modal-on-desktop pattern,
  same z-index scheme (scrim z:190, sheet z:191), same iOS Safari mandatory rules as Phase 26/28
  (no inline `display` on responsive wrappers, `useVisualViewport` keyboard tracking not needed here since
  these sheets contain no text inputs, scroll-lock while open, `onMouseDown={preventDefault}` on the close
  button to avoid the blur-before-click trap).
- Block dividers and benchmark-highlight rows are **not tappable** as dividers — only `TrainingLogEntry` rows
  (strength/run/benchmark) open a drill-down; a `BenchmarkDrilldownSheet` opens when tapping a benchmark
  entry specifically.

### Nutrition macro chip + chart (D-14, D-15, D-16)

- Tapping a `MacroChipRow` chip swaps which series `MacroTrendChart` renders (single-select, not
  multi-series-overlay) — the target reference line updates to that metric's target value from
  `UserProfileStore.nutrition_targets`.
- Tapping a bar/point on `MacroTrendChart` or a cell on `SlotAdherenceGrid` opens `DayDrilldownSheet` for
  that date — same sheet pattern as the training drill-downs.
- Slot labels throughout (chart tooltips, grid axis, day drill-down) are rendered as slot names only
  (e.g. "Post-lift", "Evening") — never formatted as or implied to be clock times (CLAUDE.md invariant,
  D-13/D-16).

### Sleep & recovery pipeline-not-live guard (D-06-style, new for this phase)

- If the `/api/health/sleep` (or equivalent) endpoint indicates the `daily_biometrics` pipeline has never
  populated any row at all (distinct from "no rows in the selected range"), render the pipeline-not-live
  placeholder copy instead of the normal per-chart empty state — this guards the documented deploy
  dependency (biometric-sync cron may not be live yet, per 30-CONTEXT.md canonical refs). Use the existing
  `PlaceholderCard` component pattern (no shimmer — this is a stable "not available yet" state, not a
  loading state).

### Backend read pattern (discretion, informs but does not gate the frontend contract)

- All three pages' endpoints follow the `fetch_nutrition_trend` / `/api/today` precedent: server computes
  aggregates (weekly bucketing, averages, `missing_dates`-equivalent gap markers), the client only renders —
  never re-derive totals or averages client-side from raw per-document reads.

## Responsive Layout

### Desktop (≥ 768px / Tailwind `md:`)

- Health renders in the same center column as Today/Tasks: `[64px sidebar] | [Health content — flex-1] |
  [GlanceRail 280px] | [DockChat 360px collapsible]` — **no full-width exception** (D-03).
- `SubTabs`: full width of the content column, 32px height, sits below whatever top padding the column
  normally uses (16px).
- Training page: two `ChartCard`s render side by side (2-column grid, `gap: 16px`) above the log list.
- Nutrition page: `MacroChipRow` + `MacroTrendChart` full width; `SlotAdherenceGrid` full width below it,
  showing its full column range without horizontal scroll if it fits ≥ 480px content width, otherwise
  scrolls horizontally within its own card (same as `ContributionGrid`'s desktop behavior).
- Sleep page: `HeaderStatRow` renders as a single inline row (no scroll needed); three chart cards stack
  vertically (D-17 — not side by side, recovery is read as a vertical narrative).
- Drill-down sheets: centered modal, max-width 480px (strength/run drill-downs with tables may use up to
  560px if the table needs more horizontal room), `border-radius: 16px`, `#0A0A0A` scrim at 70% opacity —
  same as `HabitDetailView`/`TaskDetailSheet`.

### Phone (< 768px)

- `SubTabs` sits at the top of the scrollable content area (not sticky — content is short enough per range
  that sticking is unnecessary, unlike the habit slot-group headers).
- Training page: `ChartCard`s stack vertically (chart height 160px each); log list scrolls below.
- Nutrition page: chip row scrolls horizontally if it overflows narrow screens (`overflow-x: auto`,
  `flex-shrink: 0` per chip); `SlotAdherenceGrid` scrolls horizontally within its card, auto-scrolled to the
  most recent day on mount (mirrors `ContributionGrid`'s `scrollLeft = scrollWidth` behavior).
- Sleep page: `HeaderStatRow` is a horizontal scroll strip (exact `NutritionStrip` pattern reuse).
- Drill-down sheets: bottom sheet, slides up 250ms ease-out, drag handle, scrim z:190, sheet z:191 — same as
  `TaskDetailSheet`/`HabitDetailView`.
- BottomTabs remains fixed at 64px + safe-area; `hub-main` padding-bottom convention applies unchanged.

**Invariant (carried forward, Phase 27 UAT lesson):** every wrapper whose visibility differs between phone
and desktop MUST use a Tailwind class for the `display` property (e.g. `md:hidden`, `hidden md:flex`),
never `style={{ display }}`.
